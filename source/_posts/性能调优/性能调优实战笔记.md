## 1.系统级配置优化
### （1）ulimit 设置文件描述符限制
- 每个 TCP 连接需要一个文件描述符：
	- 默认 ulimit -n 通常是 1024
	- 10000 并发至少需要 10000+ 个 fd
	- 可以设置为 65535 满足大多数测试需求
```bash
# 查看所有限制
ulimit -a

# 查看/设置文件描述符限制（最重要）
ulimit -n        # 查看
ulimit -n 65535  # 设置

# 查看/设置最大进程数
ulimit -u        # 查看
ulimit -u 65535  # 设置

# 查看/设置栈大小（KB）
ulimit -s

# 查看/设置虚拟内存（KB）
ulimit -v
```

#### <1>TCP连接参数优化

```bash

sudo sysctl -w net.ipv4.tcp_max_syn_backlog=65535           # 半连接队列最大长度（SYN_RECV状态）
sudo sysctl -w net.ipv4.tcp_fin_timeout=10                  # FIN_WAIT_2 状态超时时间（秒）
sudo sysctl -w net.ipv4.tcp_tw_reuse=1                      # 允许复用 TIME_WAIT 状态的连接

# 应用配置
```

##### 1）net.core.somaxconn=65535
```bash
作用：全连接队列（Accept Queue）的最大长度上限

全连接队列是什么？
  三次握手完成后，连接进入 ESTABLISHED 状态
  → 放入全连接队列，等待应用程序调用 accept() 取走

实际队列长度 = min(somaxconn, listen() 的 backlog 参数)

  // 应用代码
  listen(sockfd, 128);    ← backlog = 128
  
  实际长度 = min(65535, 128) = 128
  → 所以调大 somaxconn 后，应用代码的 backlog 也要调大！
  → Nginx: listen 80 backlog=65535;

队列满了会怎样？
  新完成三次握手的连接被直接丢弃（RST 或静默丢弃）
  客户端会看到 "Connection reset" 或超时

什么时候会满？
  accept() 太慢（应用处理不过来）
  突发大量并发连接（秒杀、突发流量）

监控命令：
  ss -lnt    ← 看 Send-Q（队列上限）和 Recv-Q（当前积压）
  
  State    Recv-Q  Send-Q  Local Address:Port
  LISTEN   0       128     *:80                ← Send-Q=128 是上限
  LISTEN   129     128     *:80                ← Recv-Q > Send-Q → 溢出！
```


##### 2）net.core.netdev_max_backlog=65535
```bash
作用：网卡接收队列（per-CPU backlog queue）的最大长度

注意：这个和"全连接队列"不是一回事！

                  网卡硬件
                    │
                    ↓ 中断/NAPI
            ┌───────────────┐
            │  Ring Buffer   │  ← 网卡驱动层的环形缓冲区
            └───────┬───────┘
                    ↓
            ┌───────────────┐
            │  per-CPU       │  ← netdev_max_backlog 控制这里
            │  backlog queue │     内核还没来得及处理的包暂存在此
            └───────┬───────┘
                    ↓
              协议栈处理（TCP/IP）

什么时候会满？
  网卡收包速度 > 内核协议栈处理速度
  高流量场景（10Gbps+），CPU 来不及处理

满了会怎样？
  直接丢包！在协议栈之前就丢了
  → 应用层无感知，只能通过内核计数器发现

监控：
  cat /proc/net/softnet_stat
  第二列 = 因 backlog 满而丢弃的包数
  
  netstat -s | grep "packets dropped"

```


##### 3）net.ipv4.tcp_max_syn_backlog=65535
```bash
作用：半连接队列（SYN Queue）的最大长度

半连接队列是什么？
  收到 SYN，回了 SYN+ACK，等待对方 ACK
  此时连接处于 SYN_RECV 状态 → 放在半连接队列中

  客户端 ── SYN ──→ 服务端
                     ↓
              放入半连接队列（SYN_RECV）
                     ↓
  客户端 ←─ SYN+ACK ─ 服务端
                     
  等待 ACK ...

队列满了会怎样？
  新来的 SYN 被丢弃 → 客户端看到连接超时
  → 这就是 SYN Flood 攻击的原理！

  SYN Flood 攻击：
    攻击者发大量伪造源 IP 的 SYN 包
    → 半连接队列被填满
    → 正常用户无法建立连接

防御措施（配合使用）：
  net.ipv4.tcp_syncookies = 1   ← 开启 SYN Cookie
  
  SYN Cookie 原理：
    半连接队列满后，不再存 SYN_RECV 状态
    而是把连接信息编码到 SYN+ACK 的序列号中
    收到 ACK 时解码验证 → 直接进入 ESTABLISHED
    → 不占半连接队列空间！

监控：
  netstat -s | grep "SYNs to LISTEN"
  cat /proc/net/tcp   （状态 SYN_RECV = 0x03）

```

##### 4）net.ipv4.tcp_fin_timeout=10
```bash
作用：FIN_WAIT_2 状态的超时时间

默认值：60 秒
设为：10 秒

FIN_WAIT_2 是什么？

  主动关闭方（通常是服务端）：

  服务端             客户端
    │── FIN ──→│
    │          │  服务端进入 FIN_WAIT_1
    │←── ACK ──│
    │          │  服务端进入 FIN_WAIT_2  ← 等对方发 FIN
    │          │
    │  如果客户端迟迟不发 FIN（程序bug/崩溃）...
    │  连接卡在 FIN_WAIT_2 状态
    │  
    │  tcp_fin_timeout 秒后 → 内核强制回收这个连接

为什么要调小？
  FIN_WAIT_2 的连接占用内核资源（文件描述符、内存）
  高并发服务器上可能积累大量 FIN_WAIT_2 连接
  60 秒太长 → 10 秒足够正常客户端完成关闭

风险：
  如果网络延迟大（跨国连接），10 秒可能太短
  → 连接被提前回收 → 对方发的 FIN 到了却找不到连接 → RST
  → 一般内网或低延迟环境设 10~15 秒没问题

```


##### 5）net.ipv4.tcp_tw_reuse=1
```bash
作用：允许复用 TIME_WAIT 状态的连接（仅对 客户端/主动连接方 生效）

TIME_WAIT 是什么？
  主动关闭连接的一方，发完最后一个 ACK 后进入 TIME_WAIT
  默认等待 2MSL（通常 60 秒）后才释放

  为什么要等 2MSL？
    ① 确保最后一个 ACK 能到达对方
    ② 确保网络中残留的旧数据包消亡，不会干扰新连接

问题：高并发短连接场景
  例：HTTP 短连接，服务器主动关闭
    → 每秒处理 10000 个请求
    → 每个 TIME_WAIT 持续 60 秒
    → 最多积累 600,000 个 TIME_WAIT 连接！
    → 端口耗尽，无法建立新连接

tcp_tw_reuse = 1 的效果：
  允许新的出站连接（connect()）复用 TIME_WAIT 状态的端口
  前提：TCP 时间戳选项开启（tcp_timestamps = 1，默认开启）
  
  复用条件：
    新连接的时间戳 > 旧连接的时间戳 → 可以安全区分新旧数据包

注意事项：
  ✅ 只影响主动连接方（客户端角色、反向代理连后端）
  ❌ 不影响 listen 端的 accept
  ✅ 比已废弃的 tcp_tw_recycle 安全得多（NAT 环境不会出问题）
  
  tcp_tw_recycle（已废弃，Linux 4.12+ 移除）：
    在 NAT 环境下会导致大量连接失败 → 千万别用
```


##### 6）net.ipv4.ip_local_port_range="1024 65535"
```bash
作用：客户端（主动连接方）可用的临时端口范围

默认值：32768 ~ 60999（约 28000 个端口）
设为：1024 ~ 65535（约 64500 个端口）

什么时候用到临时端口？
  connect() 时，如果没有 bind() 指定端口
  → 内核自动从 ip_local_port_range 中分配一个

  例：Nginx 作为反向代理连接后端
    upstream backend { server 10.0.0.1:8080; }
    
    Nginx → connect(10.0.0.1:8080)
    内核分配临时端口 → 10.0.0.100:34567 → 10.0.0.1:8080
                            ↑ 来自 ip_local_port_range

为什么要调大？
  一个四元组 = (源IP, 源端口, 目的IP, 目的端口)
  连接同一个后端时，目的IP和目的端口固定
  → 变化的只有源端口
  → 端口范围 = 可同时建立的最大连接数上限

  默认 28000 个端口：
    算上 TIME_WAIT 的占用，可能不够
    每秒 1000 个短连接 × 60秒 TIME_WAIT = 60000 → 超了！

  调到 64500 个端口：
    多了一倍多的余量

  还不够怎么办？
    ① 多个源 IP（增加四元组组合数）
    ② 长连接 / 连接池（减少连接建立次数）
    ③ tcp_tw_reuse = 1（复用 TIME_WAIT 端口）← 配合使用

注意：
  1024 以下是知名端口（SSH:22, HTTP:80 等）
  设 1024 起始不会和 listen 端口冲突（内核会自动避让已绑定的端口）
```

#### <2>TCP内存参数优化
```bash
sudo sysctl -w net.core.rmem_max=16777216				# 单个 socket 接收缓冲区最大值

# 应用配置

```

##### 1）net.core.rmem_max=16777216
**含义: 单个 socket 接收缓冲区最大值（字节）**
```bash
网络数据流入:

网卡 ──► 内核协议栈 ──► [ Socket 接收缓冲区 (rmem) ] ──► 应用程序 recv()
                              │
                              ▼
                    最大可设置为 rmem_max (16MB)
```
- 应用程序调用 setsockopt(SO_RCVBUF) 时的上限
- **默认值**: 约 208KB
- **16777216** = 16MB
- **用途**: 高带宽、高延迟网络需要更大缓冲区

##### 2）net.core.wmem_max=16777216
**含义: 单个 socket 发送缓冲区最大值（字节）**
```bash
网络数据流出:

应用程序 send() ──► [ Socket 发送缓冲区 (wmem) ] ──► 内核协议栈 ──► 网卡
                              │
                              ▼
                    最大可设置为 wmem_max (16MB)

```
- 应用程序调用 setsockopt(SO_SNDBUF) 时的上限
- **默认值**: 约 208KB
- **16777216** = 16MB
- **用途**: 大文件传输、流媒体服务

##### 3）net.ipv4.tcp_rmem="4096 87380 16777216"
**含义: TCP 接收缓冲区自动调节范围**
```bash
格式: "min default max"
       │    │      │
       │    │      └── 最大值: 16777216 (16MB)
       │    └── 默认值: 87380 (~85KB)
       └── 最小值: 4096 (4KB)
```
**工作原理:**
```bash
┌─────────────────────────────────────────────────────────────┐
│                    TCP 缓冲区自动调节                        │
│                                                             │
│   低流量         中等流量           高流量/高延迟             │
│     │              │                   │                    │
│     ▼              ▼                   ▼                    │
│   4KB ─────────► 85KB ─────────────► 16MB                   │
│   (min)        (default)            (max)                   │
│                                                             │
│   内核会根据：                                               │
│   - 网络带宽                                                 │
│   - 往返延迟 (RTT)                                          │
│   - 可用内存                                                 │
│   自动调整每个连接的缓冲区大小                                │
└─────────────────────────────────────────────────────────────┘
```

**含义: TCP 发送缓冲区自动调节范围**

```bash
       │    │      └── 最大值: 16777216 (16MB)
```

## 2.池化技术详解与调优
### （1）未使用池化技术时的问题
```bash
每次请求都要：

┌─────────┐     创建      ┌─────────┐     使用      ┌─────────┐     销毁      ┌─────────┐
│  请求1   │ ──────────► │  资源1   │ ──────────► │  处理    │ ──────────► │  释放    │
└─────────┘   (耗时)      └─────────┘              └─────────┘   (耗时)      └─────────┘

问题：
• 创建/销毁开销大（线程创建 ~1ms，连接建立 ~10-100ms）
• 频繁 GC / 内存碎片
• 资源数量不可控（可能耗尽系统资源）
```



### （2）四种常用池化技术详解
#### <1>连接池
```bash
┌────────────────────────────────────────────────────────────┐
│                      连接池                                 │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐                  │
│  │ 空闲 │ │ 空闲 │ │使用中│ │使用中│ │ 空闲 │                  │
│  │ DB连接│ │Redis│ │ DB  │ │Redis│ │ DB  │                  │
│  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘                  │
│                                                            │
│  • 避免频繁 TCP 三次握手/四次挥手                             │
│  • 避免频繁 MySQL 认证、Redis AUTH                          │
│  • 复用长连接，减少 TIME_WAIT                                │
└────────────────────────────────────────────────────────────┘

应用场景：
├── 数据库连接（MySQL、PostgreSQL）
```

#### <3>线程池
- **注意区分“CPU 密集型任务” 与 “IO 密集型任务”**
```bash
┌────────────────────────────────────────────────────────────┐
│                      线程池                                 │
│                                                            │
│   任务队列                    工作线程                       │
│  ┌──────────┐               ┌────────┐                     │
│  │ Task 1   │               │Thread 1│ ◄── 处理中          │
│  │ Task 2   │               │Thread 2│ ◄── 处理中          │
│  │ Task 3   │ ────等待────► │Thread 3│ ◄── 空闲            │
│  │ Task 4   │               │Thread 4│ ◄── 处理中          │
│  │  ...     │               │  ...   │                     │
│  └──────────┘               └────────┘                     │
│                                                            │
│  • 避免频繁创建/销毁线程的开销                                │
│  • 控制并发线程数量，防止资源耗尽                             │
│  • 任务排队，削峰填谷                                        │
└────────────────────────────────────────────────────────────┘

线程数设置公式：

CPU 密集型任务：线程数 = CPU 核心数（或 +1）
IO 密集型任务： 线程数 = CPU 核心数 × (1 + IO等待时间/CPU计算时间)

示例：
├── 纯计算任务（加密、压缩）：8核 → 8线程
├── Web 服务器（大量 IO）：  8核 → 16-32 线程
└── 混合型任务：            8核 → 12-16 线程
```
