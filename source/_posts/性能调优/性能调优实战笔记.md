# 二、性能调优方案

## 1.系统级配置优化
### （1）ulimit 设置文件描述符限制
- 每个 TCP 连接需要一个文件描述符：
	- 默认 ulimit -n 通常是 1024
	- 10000 并发至少需要 10000+ 个 fd
	- 可以设置为 65535 满足大多数测试需求
```bash
# 查看所有限制
ulimit -a

# 查看/设置文件描述符限制（最重要）
ulimit -n        # 查看
ulimit -n 65535  # 设置

# 查看/设置最大进程数
ulimit -u        # 查看
ulimit -u 65535  # 设置

# 查看/设置栈大小（KB）
ulimit -s

# 查看/设置虚拟内存（KB）
ulimit -v
```

#### <1>TCP连接参数优化

```bash
sudo sysctl -w net.core.somaxconn=65535						# 监听队列最大长度
sudo sysctl -w net.core.netdev_max_backlog=65535			# 网卡接收队列最大长度
sudo sysctl -w net.ipv4.tcp_max_syn_backlog=65535			# 半连接队列最大长度（SYN_RECV状态）
sudo sysctl -w net.ipv4.tcp_fin_timeout=10					# FIN_WAIT_2 状态超时时间（秒）
sudo sysctl -w net.ipv4.tcp_tw_reuse=1						# 允许复用 TIME_WAIT 状态的连接
sudo sysctl -w net.ipv4.ip_local_port_range="1024 65535"	# 本地端口分配范围

# 应用配置
sudo sysctl -p
```

##### 1）net.core.somaxconn=65535
**含义: 监听队列最大长度**
```bash
客户端连接请求 ──► [ 监听队列 (somaxconn) ] ──► accept() 处理
```
- 当服务器调用 listen(fd, backlog) 时，backlog 的上限就是 somaxconn
- 队列满了之后，新连接会被丢弃或收到 RST
- **默认值**: 128（太小！）
- **高并发建议**: 65535

##### 2）net.core.netdev_max_backlog=65535
**含义: 网卡接收队列最大长度**
```bash
网卡收到数据包 ──► [ 网卡队列 (netdev_max_backlog) ] ──► CPU 处理
```
- 当 CPU 处理不过来时，数据包会在这个队列中排队
- 队列满了，新数据包会被丢弃
- **默认值**: 1000
- **高并发建议**: 65535（配合多核 CPU）

##### 3）net.ipv4.tcp_max_syn_backlog=65535
**含义: 半连接队列最大长度（SYN_RECV 状态）**
```bash
TCP 三次握手:

客户端 ────► SYN ────► 服务器
             │
             ▼
      [ 半连接队列 (tcp_max_syn_backlog) ]  ◄── SYN_RECV 状态
             │
客户端 ◄─── SYN+ACK ◄── 服务器
             │
客户端 ────► ACK ────► 服务器
             │
             ▼
      [ 全连接队列 (somaxconn) ]  ◄── ESTABLISHED 状态
```
- 存放已收到 SYN 但未完成三次握手的连接
- 防止 SYN Flood 攻击时队列溢出
- **默认值**: 256
- **高并发建议**: 65535

##### 4）net.ipv4.tcp_fin_timeout=10
**含义: FIN_WAIT_2 状态超时时间（秒）**
```bash
TCP 四次挥手:

主动关闭方                    被动关闭方
    │                            │
    │──── FIN ──────────────────►│
    │     (FIN_WAIT_1)           │
    │◄─────── ACK ───────────────│
    │     (FIN_WAIT_2) ◄── 在这里等待
    │                            │
    │◄─────── FIN ───────────────│
    │──── ACK ──────────────────►│
    │     (TIME_WAIT)            │
```
- 主动关闭连接方在 FIN_WAIT_2 状态等待对方 FIN 的最长时间
- 超时后强制关闭，释放资源
- **默认值**: 60（太长！）
- **高并发建议**: 10-30（快速回收）

##### 5）net.ipv4.tcp_tw_reuse=1
**含义: 允许复用 TIME_WAIT 状态的连接**
```bash
TIME_WAIT 状态:
┌─────────────────────────────────────────────┐
│  连接关闭后，端口会进入 TIME_WAIT 状态        │
│  默认等待 2*MSL（约 60 秒）才能复用          │
│                                             │
│  问题: 高并发时大量端口被 TIME_WAIT 占用      │
│  解决: tcp_tw_reuse=1 允许复用这些端口        │
└─────────────────────────────────────────────┘
```
- **0**: 禁止复用（默认）
- **1**: 允许复用（客户端主动连接时生效）
- **高并发建议**: 1

##### 6）net.ipv4.ip_local_port_range="1024 65535"
**含义: 本地端口分配范围**
```bash
客户端发起连接时，系统自动分配源端口:

可用端口范围: [1024] ──────────────────► [65535]
              │                           │
              └─── 共 64511 个端口可用 ────┘
```
- 每个 TCP 连接需要一个唯一的 (源IP, 源端口, 目标IP, 目标端口) 四元组
- 端口范围越大，支持的并发连接数越多
- **默认值**: "32768 60999"（约 28000 个端口）
- **高并发建议**: "1024 65535"（约 64000 个端口）

#### <2>TCP内存参数优化
```bash
sudo sysctl -w net.core.rmem_max=16777216				# 单个 socket 接收缓冲区最大值
sudo sysctl -w net.core.wmem_max=16777216				# 单个 socket 发送缓冲区最大值
sudo sysctl -w net.ipv4.tcp_rmem="4096 87380 16777216"	# TCP 接收缓冲区自动调节范围
sudo sysctl -w net.ipv4.tcp_wmem="4096 65536 16777216"	# TCP 发送缓冲区自动调节范围

# 应用配置
sudo sysctl -p
```

##### 1）net.core.rmem_max=16777216
**含义: 单个 socket 接收缓冲区最大值（字节）**
```bash
网络数据流入:

网卡 ──► 内核协议栈 ──► [ Socket 接收缓冲区 (rmem) ] ──► 应用程序 recv()
                              │
                              ▼
                    最大可设置为 rmem_max (16MB)
```
- 应用程序调用 setsockopt(SO_RCVBUF) 时的上限
- **默认值**: 约 208KB
- **16777216** = 16MB
- **用途**: 高带宽、高延迟网络需要更大缓冲区

##### 2）net.core.wmem_max=16777216
**含义: 单个 socket 发送缓冲区最大值（字节）**
```bash
网络数据流出:

应用程序 send() ──► [ Socket 发送缓冲区 (wmem) ] ──► 内核协议栈 ──► 网卡
                              │
                              ▼
                    最大可设置为 wmem_max (16MB)

```
- 应用程序调用 setsockopt(SO_SNDBUF) 时的上限
- **默认值**: 约 208KB
- **16777216** = 16MB
- **用途**: 大文件传输、流媒体服务

##### 3）net.ipv4.tcp_rmem="4096 87380 16777216"
**含义: TCP 接收缓冲区自动调节范围**
```bash
格式: "min default max"
       │    │      │
       │    │      └── 最大值: 16777216 (16MB)
       │    └── 默认值: 87380 (~85KB)
       └── 最小值: 4096 (4KB)
```
**工作原理:**
```bash
┌─────────────────────────────────────────────────────────────┐
│                    TCP 缓冲区自动调节                        │
│                                                             │
│   低流量         中等流量           高流量/高延迟             │
│     │              │                   │                    │
│     ▼              ▼                   ▼                    │
│   4KB ─────────► 85KB ─────────────► 16MB                   │
│   (min)        (default)            (max)                   │
│                                                             │
│   内核会根据：                                               │
│   - 网络带宽                                                 │
│   - 往返延迟 (RTT)                                          │
│   - 可用内存                                                 │
│   自动调整每个连接的缓冲区大小                                │
└─────────────────────────────────────────────────────────────┘
```

##### 4）net.ipv4.tcp_wmem="4096 65536 16777216"
**含义: TCP 发送缓冲区自动调节范围**
```bash
格式: "min default max"
       │    │      │
       │    │      └── 最大值: 16777216 (16MB)
       │    └── 默认值: 65536 (64KB)
       └── 最小值: 4096 (4KB)
```
**发送缓冲区工作流程:**
```bash
应用程序 write()/send()
         │
         ▼
┌─────────────────────────────────────┐
│        TCP 发送缓冲区                │
│  ┌─────┬─────┬─────┬─────┬─────┐   │
│  │已确认│ 已发送 │ 待发送 │  空闲  │   │
│  │可删除│未确认  │       │       │   │
│  └─────┴─────┴─────┴─────┴─────┘   │
│                                     │
│  缓冲区满 → write() 阻塞或返回 EAGAIN │
└─────────────────────────────────────┘
         │
         ▼
      网络发送
```

## 2.池化技术详解与调优
### （1）未使用池化技术时的问题
```bash
每次请求都要：

┌─────────┐     创建      ┌─────────┐     使用      ┌─────────┐     销毁      ┌─────────┐
│  请求1   │ ──────────► │  资源1   │ ──────────► │  处理    │ ──────────► │  释放    │
└─────────┘   (耗时)      └─────────┘              └─────────┘   (耗时)      └─────────┘

问题：
• 创建/销毁开销大（线程创建 ~1ms，连接建立 ~10-100ms）
• 频繁 GC / 内存碎片
• 资源数量不可控（可能耗尽系统资源）
```



### （2）四种常用池化技术详解
#### <1>连接池
```bash

应用场景：
├── 数据库连接（MySQL、PostgreSQL）

├── HTTP 连接（HTTP Keep-Alive）
```


#### <3>线程池
- **注意区分“CPU 密集型任务” 与 “IO 密集型任务”**
```bash
┌────────────────────────────────────────────────────────────┐
│                      线程池                                 │
│                                                            │
│   任务队列                    工作线程                       │
│  ┌──────────┐               ┌────────┐                     │
│  │ Task 1   │               │Thread 1│ ◄── 处理中          │
│  │ Task 2   │               │Thread 2│ ◄── 处理中          │
│  │ Task 3   │ ────等待────► │Thread 3│ ◄── 空闲            │
│  │ Task 4   │               │Thread 4│ ◄── 处理中          │
│  │  ...     │               │  ...   │                     │
│  └──────────┘               └────────┘                     │
│                                                            │
│  • 避免频繁创建/销毁线程的开销                                │
│  • 控制并发线程数量，防止资源耗尽                             │
│  • 任务排队，削峰填谷                                        │
└────────────────────────────────────────────────────────────┘

线程数设置公式：

CPU 密集型任务：线程数 = CPU 核心数（或 +1）
IO 密集型任务： 线程数 = CPU 核心数 × (1 + IO等待时间/CPU计算时间)

示例：
├── 纯计算任务（加密、压缩）：8核 → 8线程
├── Web 服务器（大量 IO）：  8核 → 16-32 线程
└── 混合型任务：            8核 → 12-16 线程
```

```bash
│                                                            │
│  获取对象：从池中取出 → 重置状态 → 使用                        │

│                                                            │
│  • 避免频繁 new/delete                                      │
│  • 减少 GC 压力（Java/Go）                                  │
│  • 对象复用，减少初始化开销                                   │
└────────────────────────────────────────────────────────────┘

应用场景：
├── 聊天消息对象
├── 网络请求/响应对象
├── 数据库查询结果对象
└── 游戏中的子弹、特效对象
```

### （3）池化技术配置调优
```cpp
┌─────────────────────────────────────────────────────────────────────┐
│                        池配置参数                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  容量相关：                                                          │
│  ├── 初始大小 (initialSize)      - 启动时创建的资源数量               │
│  ├── 最小空闲 (minIdle)          - 保持的最少空闲资源数               │
│  ├── 最大空闲 (maxIdle)          - 保持的最多空闲资源数               │
│  └── 最大大小 (maxSize)          - 池中最多资源总数                   │
│                                                                     │
│  超时相关：                                                          │
│  ├── 获取超时 (acquireTimeout)   - 获取资源的最大等待时间             │
│  ├── 空闲超时 (idleTimeout)      - 空闲资源被回收的时间               │
│  ├── 最大存活 (maxLifetime)      - 资源最大存活时间（防止泄漏）        │
│  └── 连接超时 (connectionTimeout)- 创建新资源的超时时间               │
│                                                                     │
│  重试相关：                                                          │
│  ├── 最大重试次数 (maxRetries)   - 获取失败后重试次数                 │
│  ├── 重试间隔 (retryInterval)    - 重试之间的等待时间                 │
│  └── 重试退避 (backoffMultiplier)- 指数退避因子                      │
│                                                                     │
│  健康检查：                                                          │
│  ├── 检查间隔 (validationInterval)- 定期检查资源有效性               │
│  ├── 借出时检查 (testOnBorrow)    - 获取时是否验证                   │
│  ├── 归还时检查 (testOnReturn)    - 归还时是否验证                   │
│  └── 空闲时检查 (testWhileIdle)   - 空闲时是否验证                   │
│                                                                     │
│  扩缩容：                                                            │
│  ├── 扩容步长 (acquireIncrement) - 每次扩容创建的资源数               │
│  ├── 缩容检查间隔 (shrinkInterval)- 检查是否需要缩容的间隔            │
│  └── 预热 (warmup)               - 启动时是否预创建资源               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```
