# 一、异常处理三层模型
- 作为一个新手，我时常对**异常的抛出与捕捉**感到疑惑：
    - 什么时候应该抛出异常？
    - 抛出异常时是否需要输出日志？
    - 应该抛出什么类型的异常？
    - 什么时候需要定义自己的异常类？为什么要定义具体的异常类？
    - 什么时候应该捕获异常？捕获异常后是直接输出日志，还是继续向上层抛出？
- 下面我将给出我的思考，**仅供参考**

## 1.三层结构（底层——>上层）
- **C语言第三方库：**
    - C 语言第三方库没有异常类的概念，只能通过错误码和错误信息（errno、error）来表示异常
- **C++封装层：**
    - 对 C 语言第三方库进行C++封装，针对 C 语言的错误码定义相应的异常类
- **C++使用层**
    - 使用封装好的C++代码，捕获异常，进行日志输出，返回Result等（尽可能不再抛出异常）

```cpp
                ▼

    │   catch 异常          │  ← 记日志 + 转换为 Result
    │   LOG_ERROR(...)      │
    │   return Result       │  ← 返回 Result，不抛异常
    │                       │
    │   （可多层嵌套）       │  ← 层层传递 Result
    └───────────────────────┘
```

### （1）C语言第三方库
- 由于 C 语言没有类、异常抛出与捕获的概念，因此 C 语言第三方库在处理异常时，通常是通过（errno, error）来反映所产生的异常。这种方式处理起来相对繁琐

### （2）C++封装层
- **组成**：
    - C++ 第三方库
    - 自己对 C 语言库 API 进行 C++ 封装后生成的库等
- 许多 C++ 第三方库本质上都是对 C 语言第三方库 API 的封装。封装后的 API 不再返回（errno, error），而是**自定义"异常类"**，并在 API 发生异常时通过 throw 抛出给上层，方便上层捕获并处理异常
- 除了使用第三方库之外，有时我们也需要自己封装 C 语言第三方库，例如：自己封装 MySQL C 库的 API
    - 这时，你就需要针对 C 语言第三方库的（errno, error）自定义相应的"服务端异常类"，这样才能方便上层调用
    - 同时：定义具体的异常类，上层捕捉后才能做出相应的处理
- **主要职责**：
    - **定义服务端异常**
    - **只抛出异常，不进行日志输出**
        - 为 C++ 封装层相对于上层来说属于底层库，只需要将异常信息填入异常类中并抛出给上层使用者即可。至于使用者如何处理，那是使用者自己的事情
        - C++ 封装层只抛出异常，就像 C 库只返回（errno、error）一样，都不输出日志，而是让调用者自己决定如何处理
```cpp
// C++ 封装层
void MySQLConnection::Query(const std::string& sql) {
    if (mysql_query(...) != 0) {
        // ❌ 不要在这里 LOG_ERROR
        // 因为：
        // 1. 上层可能会重试，这不算"真正的错误"
        // 2. 上层可能故意触发这个错误（如检查用户是否存在）
        // 3. 同一个异常可能被记录多次
        throw MySQLQueryException(mysql_error(mysql_));
    }
}
```

### （3）C++使用层
- 在这一层，我们使用封装好的 C++ 库（如 redis++、hiredis 封装等），可能会接收到各种异常
- **主要职责**：
    - **定义“错误码”、“Result”**
    - **捕获异常，进行日志输出，创建对应的Result作为返回值**
    - **尽可能不再抛出异常**
- **注意**：
    - 这"一层"在开发中并不是只有一层，可能是**层层封装**，但这一层几乎都**不再抛出异常**，统一返回 Result，可能要进行 Result 之间的转换
    - **某些特殊API还是要抛出异常**
        - 构造函数、重载函数等
        - init等初始化函数等

```cpp
// C++ 使用层 - 第一层：Repository（直接使用 C++ 封装层）
Result<int64_t> UserRepository::Create(const std::string& username, const std::string& password) {
    try {
        auto conn = pool_->Get();
        conn->Execute("INSERT INTO users ...", {username, password});
        return Result<int64_t>::Ok(conn->LastInsertId());
        
    } catch (const MySQLConflictException& e) {
        // 重复键：业务结果，不是系统错误
        LOG_WARN("用户已存在: {}, 原因: {}", username, e.what());
        return Result<int64_t>::Fail(ErrorCode::USER_ALREADY_EXISTS, "用户名已被注册");
        
    } catch (const MySQLRetryableException& e) {
        // 可重试错误 → 服务暂时不可用
        LOG_ERROR("数据库连接异常: {}", e.what());
        return Result<int64_t>::Fail(ErrorCode::SERVICE_UNAVAILABLE, "服务暂时不可用");
        
    } catch (const MySQLException& e) {
        // 其他数据库错误 → 内部错误
        LOG_ERROR("创建用户失败: {}", e.what());
        return Result<int64_t>::Fail(ErrorCode::INTERNAL_ERROR, "系统内部错误");
    }
}
```

#### <3>错误信息处理原则
- 对于后端错误（如资源不足、内部异常等），返回给客户端的错误信息应填写"服务器繁忙"、"服务暂不可用"等模糊表达
    - 一方面：内部异常信息客户端是看不懂的
    - 另一方面：暴露内部错误信息（如 SQL 语句、表名、字段名）可能造成安全风险
    - 详细错误信息只在日志中记录，供开发人员排查问题
```cpp
// ❌ 错误示例：暴露内部信息
return Result::Fail(ErrorCode::DATABASE_ERROR, 
    "MySQL error 1062: Duplicate entry 'admin' for key 'users.username'");

// ✅ 正确示例：模糊处理
LOG_ERROR("创建用户失败: MySQL error 1062: Duplicate entry ...");  // 日志记录详细信息
return Result::Fail(ErrorCode::USER_ALREADY_EXISTS, "用户名已被注册");  // 返回用户友好信息
```

### 第1层：模拟 C 语言库
```cpp
// c_mysql_lib.h
#pragma once
#include <cstring>

namespace c_mysql {

// 模拟 C 库的全局错误状态
inline int g_errno = 0;
inline const char* g_error = "";

// 模拟执行 SQL（C 风格 API）
inline int mysql_query(const char* sql) {
    if (strstr(sql, "duplicate") != nullptr) {
        g_errno = 1062;
        g_error = "Duplicate entry for key 'username'";
        return -1;
    }
    if (strstr(sql, "timeout") != nullptr) {
        g_errno = 2006;
        g_error = "MySQL server has gone away";
        return -1;
    }
    g_errno = 0;
    g_error = "";
    return 0;  // 成功
}

inline int mysql_errno() { return g_errno; }
inline const char* mysql_error() { return g_error; }

}  // namespace c_mysql
```

### 第2层：C++ 封装层（抛异常，不记日志）
```cpp
// mysql_wrapper.h
#pragma once
#include "c_mysql_lib.h"
#include <stdexcept>
#include <string>

namespace db {

// ==================== 异常类定义 ====================
class MySQLException : public std::runtime_error {
public:
    int code;
    
    MySQLException(int c, const std::string& msg) 
        : std::runtime_error(msg), code(c) {}
    
    // 提供错误码访问方法
    int GetErrorCode() const noexcept { return code; }
    
    // 虚析构函数（虽然 runtime_error 已有，但显式声明更清晰）
    ~MySQLException() override = default;
};

// 重复键异常（如：用户名已存在）
class MySQLDuplicateException : public MySQLException {
public:
    using MySQLException::MySQLException;
};

// 连接异常（可重试）
class MySQLConnectionException : public MySQLException {
public:
    using MySQLException::MySQLException;
};

// ==================== 封装类 ====================
class MySQLConnection {
public:
    // 执行 SQL，失败时抛出异常
    void Execute(const std::string& sql) {
        int ret = c_mysql::mysql_query(sql.c_str());
        
        if (ret != 0) {
            int code = c_mysql::mysql_errno();
            std::string msg = c_mysql::mysql_error();
            
            // ❌ 不在这里记日志！让上层决定如何处理
            // ✅ 根据错误码抛出不同异常
            if (code == 1062) {
                throw MySQLDuplicateException(code, msg);
            } else if (code == 2006 || code == 2013) {
                throw MySQLConnectionException(code, msg);
            } else {
                throw MySQLException(code, msg);
            }
        }
    }
};

}  // namespace db
```

### 通用组件：Result 定义
```cpp
// result.h
#pragma once
#include <string>
#include <iostream>

// ==================== 错误码 ====================
enum class ErrorCode {
    OK = 0,
    USER_ALREADY_EXISTS = 1001,
    SERVICE_UNAVAILABLE = 5001,
    INTERNAL_ERROR      = 5002,
};

// ==================== Result 类型 ====================
template<typename T>
struct Result {
    ErrorCode code;
    std::string message;
    T data;
    
    // Result 类中，不会抛异常的方法应标记 noexcept
    bool Success() const noexcept { return code == ErrorCode::OK; }
    bool Failed() const noexcept { return !Success(); }
    
    // 成功
    static Result Ok(T d) { 
        return {ErrorCode::OK, "success", d}; 
    }
    
    // 失败
    static Result Fail(ErrorCode c, const std::string& msg) { 
        return {c, msg, T{}}; 
    }
    
    // 打印
    void Print() const {
        std::cout << "[Result] code=" << (int)code 
                  << ", message=" << message << std::endl;
    }
};

// void 特化
template<>
struct Result<void> {
    ErrorCode code;
    std::string message;
    
    bool Success() const noexcept { return code == ErrorCode::OK; }
    bool Failed() const noexcept { return !Success(); }
    
    static Result Ok() { return {ErrorCode::OK, "success"}; }
    static Result Fail(ErrorCode c, const std::string& msg) { return {c, msg}; }
    
    void Print() const {
        std::cout << "[Result] code=" << (int)code 
                  << ", message=" << message << std::endl;
    }
};
```

### 第3层：C++ 使用层（捕获异常，返回 Result）
```cpp
// user_repository.h
#pragma once
#include "mysql_wrapper.h"
#include "result.h"
#include <iostream>

// 简单日志宏
#define LOG_WARN(msg)  std::cerr << "[WARN] "  << msg << std::endl
#define LOG_ERROR(msg) std::cerr << "[ERROR] " << msg << std::endl

namespace service {

class UserRepository {
private:
    db::MySQLConnection conn_;
    
public:
    // 创建用户：捕获异常 → 记日志 → 返回 Result
    Result<int> CreateUser(const std::string& username) {
        try {
            // ⚠️ 示例代码，仅用于演示异常处理流程
            // 实际项目中必须使用参数化查询（Prepared Statement）防止 SQL 注入
            std::string sql = "INSERT INTO users (username) VALUES ('" + username + "')";
            conn_.Execute(sql);
            return Result<int>::Ok(1);  // 假设返回用户ID
            
        } catch (const db::MySQLDuplicateException& e) {
            // 业务错误：用户已存在
            LOG_WARN("用户已存在: " + username);
            return Result<int>::Fail(ErrorCode::USER_ALREADY_EXISTS, "用户名已被注册");
            
        } catch (const db::MySQLConnectionException& e) {
            // 系统错误：连接问题
            LOG_ERROR("数据库连接失败: " + std::string(e.what()));
            return Result<int>::Fail(ErrorCode::SERVICE_UNAVAILABLE, "服务暂时不可用");
            
        } catch (const db::MySQLException& e) {
            // 其他错误
            LOG_ERROR("数据库错误: " + std::string(e.what()));
            return Result<int>::Fail(ErrorCode::INTERNAL_ERROR, "系统错误");
        }
    }
};

// ==================== Service 层（使用 Repository）====================
class UserService {
private:
    UserRepository repo_;
    
public:
    // 注册用户：检查 Result，传递 Result
    Result<int> Register(const std::string& username) {
        // 参数校验
        if (username.empty()) {
            return Result<int>::Fail(ErrorCode::INTERNAL_ERROR, "用户名不能为空");
        }
        
        // 调用 Repository（不会抛异常，返回 Result）
        auto result = repo_.CreateUser(username);
        
        // 直接传递 Result
        return result;
    }
};

}  // namespace service
```

## 3.实战示例
- 该示例主要是对redis++ API进行业务层封装，方便业务层使用
```cpp
// redis_client.h
#pragma once

#include <sw/redis++/redis++.h>
#include <memory>
#include <string>
#include <optional>
#include <chrono>
#include <vector>
#include <unordered_map>
#include "config/config.h"
#include "common/result.h"

namespace user_service {

// ============================================================================
// Redis 客户端
// 
// 设计原则：
//   - 使用 Result<T> 区分"执行失败"与"数据不存在"
//   - 所有操作都是 noexcept，通过 Result 返回错误信息
//   - Result 成功 + optional 无值 = key 不存在（正常业务情况）
//   - Result 失败 = Redis 执行出错（网络/超时等）
//
// 使用示例：
//   auto result = redis->Get("user:123");
//   if (!result.IsOk()) {
//       // Redis 执行失败
//       LOG_ERROR("Redis error: {}", result.message);
//   } else if (result.Value().has_value()) {
//       // Key 存在，获取到值
//       auto& value = result.Value().value();
//   } else {
//       // Key 不存在（正常情况）
//   }
// ============================================================================
class RedisClient {
public:
    /// @brief 构造函数（详细参数版本）
    RedisClient(const std::string& host, int port,
                const std::string& password, int db);
    
    /// @brief 构造函数（配置结构体版本）
    explicit RedisClient(const RedisConfig& config);

    // ==================== 字符串操作 ====================
    
    /// @brief 设置键值对（无过期时间）
    /// @return 成功返回 Ok，失败返回错误信息
    Result<void> Set(const std::string& key, const std::string& value) noexcept;
    
    /// @brief 设置键值对 + 过期时间（毫秒）
    /// @param ttl 过期时间（毫秒），必须 > 0
    /// @return 成功返回 Ok，失败返回错误信息
    Result<void> SetPx(const std::string& key, const std::string& value,
                       std::chrono::milliseconds ttl) noexcept;
    
    /// @brief 获取值
    /// @return 成功+有值=存在，成功+无值=不存在，失败=Redis错误
    Result<std::optional<std::string>> Get(const std::string& key) noexcept;

    /// @brief 仅当 key 不存在时设置（分布式锁常用）
    /// @return 成功时 bool 表示是否设置成功（true=设置了，false=key已存在）
    Result<bool> SetNx(const std::string& key, const std::string& value) noexcept;

    /// @brief SETNX + PEXPIRE 原子操作（分布式锁推荐）
    Result<bool> SetNxPx(const std::string& key, const std::string& value,
                         std::chrono::milliseconds ttl) noexcept;

    // ==================== 通用操作 ====================

    Result<std::vector<std::string>> Keys(const std::string& pattern) noexcept;

    Result<bool> HExists(const std::string& key, const std::string& field) noexcept;

    // ==================== 健康检查 ====================
    
    /// @brief Ping 测试连接
```

# 二、知识补充
## 1.C++使用层并非完全不抛出异常
- 对于“**构造函数**”等“无返回值的API”，如果内部出错，还是需要抛出异常的
- 同理，**操作符重载**也无法改变返回类型，必须通过异常报告错误

```cpp
// 构造函数：无法返回 Result，只能抛异常
RedisClient::RedisClient(const RedisConfig& config) {
    try {
        redis_ = std::make_unique<sw::redis::Redis>(conn_opt, pool_opt);
    } catch (const sw::redis::Error& e) {
        LOG_ERROR("Redis connection failed: {}", e.what());
        throw;  // ✅ 构造函数中允许抛异常
    }
}

// 操作符重载：无法返回 Result
BigInt BigInt::operator+(const BigInt& other) const {
    if (/* 溢出检测 */) {
        throw std::overflow_error("BigInt overflow");  // ✅ 只能抛异常
    }
    return result;
}
```


## 2.noexcept 的使用
- 在 C++ 使用层，几乎不会抛出异常，大都以 Result 进行返回
- 因此在这一层封装的 API 应使用 noexcept 标识，有以下好处：
    - **文档作用**：明确告诉调用者"这个函数不会抛异常"
    - **编译器优化**：编译器可以生成更高效的代码
    - **异常安全**：如果 noexcept 函数真的抛出异常，程序会直接终止（std::terminate）

```cpp
// ✅ 好的实践：Result 返回的函数标记 noexcept
Result<void> Set(const std::string& key, const std::string& value) noexcept;
Result<std::optional<std::string>> Get(const std::string& key) noexcept;

// ✅ 不会抛异常的成员函数也应标记
bool Success() const noexcept { return code == ErrorCode::OK; }
int GetErrorCode() const noexcept { return code; }
```


## 3.“宏定义”减少重复代码
- Result相比于“抛出异常”的缺点就是**“重复代码”较多**，每个Result都需要检查
- 可以通过"宏定义"来减少重复代码

### （1）捕获异常的宏
```cpp
// 单 key 操作的异常捕获
#define REDIS_CATCH_RETURN(ResultType, operation, key)                         \
    catch (const sw::redis::Error& e) {                                        \
        LOG_WARN("Redis {} failed: key={}, err={}", operation, key, e.what()); \
        return ResultType::Fail(ErrorCode::ServiceUnavailable, e.what());      \
    }

// 使用示例
Result<void> RedisClient::Set(const std::string& key, const std::string& value) noexcept {
    try {
        redis_->set(key, value);
        return Result<void>::Ok();
    }
    REDIS_CATCH_RETURN(Result<void>, "SET", key)  // 展开为 catch 块
}
```

### （2）检查 Result 并提前返回的宏
```cpp
// 检查 Result，失败则提前返回（用于 Result 传递）
#define RETURN_IF_ERROR(result)           \
    do {                                  \
        if ((result).Failed()) {          \
            return result;                \
        }                                 \
    } while (0)

// 检查 Result，失败则返回转换后的类型
#define RETURN_IF_ERROR_AS(result, ReturnType)                    \
    do {                                                          \
        if ((result).Failed()) {                                  \
            return ReturnType::Fail((result).code, (result).message); \
        }                                                         \
    } while (0)

// 使用示例
Result<UserInfo> UserService::GetUserWithProfile(int64_t user_id) {
    // 获取用户基本信息
    auto user_result = repo_.GetUser(user_id);
    RETURN_IF_ERROR_AS(user_result, Result<UserInfo>);
    
    // 获取用户资料
    auto profile_result = repo_.GetProfile(user_id);
    RETURN_IF_ERROR_AS(profile_result, Result<UserInfo>);
    
    // 组合返回
    UserInfo info;
    info.user = user_result.Value();
    info.profile = profile_result.Value();
    return Result<UserInfo>::Ok(std::move(info));
}
```

### （3）解包 Result 值的宏
```cpp
// 解包 Result，失败则返回错误
#define TRY_UNWRAP(var, result, ReturnType)                          \
    auto _tmp_##var = (result);                                       \
    if (_tmp_##var.Failed()) {                                        \
        return ReturnType::Fail(_tmp_##var.code, _tmp_##var.message); \
    }                                                                 \
    auto var = std::move(_tmp_##var).Value()

// 使用示例（更简洁）
Result<UserInfo> UserService::GetUserWithProfile(int64_t user_id) {
    TRY_UNWRAP(user, repo_.GetUser(user_id), Result<UserInfo>);
    TRY_UNWRAP(profile, repo_.GetProfile(user_id), Result<UserInfo>);
    
    return Result<UserInfo>::Ok({user, profile});
}
```

## 4.“模糊信息”返回给前端
- 后端的错误信息前端用户大多是看不懂的
- 直接返回后端的错误信息会暴露内部实现，存在安全风险
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260202114239102.png)

```cpp
// ❌ 危险：暴露内部信息
return Result::Fail(ErrorCode::DATABASE_ERROR, 
    "MySQL error 1062: Duplicate entry 'admin' for key 'users.username'");
    // 攻击者可以得知：使用 MySQL、表名是 users、有 username 字段

// ✅ 安全：模糊处理
LOG_ERROR("创建用户失败: MySQL error 1062: Duplicate entry 'admin' for key 'users.username'");
return Result::Fail(ErrorCode::USER_ALREADY_EXISTS, "用户名已被注册");
```

## 5.std::expected 的使用
- C++23 引入了 std::expected<T, E>，是标准库对 Result 模式的官方支持
- 如果项目可以使用 C++23，推荐使用 std::expected 代替自定义 Result

## 6.异常捕获的顺序
- **必须从最具体的派生类到最一般的基类**
- 如果把基类放前面，派生类的 catch 永远不会执行
```cpp
// ✅ 正确顺序：派生类 → 基类
try {
    // ...
} catch (const MySQLDuplicateException& e) {   // 1. 最具体（派生类）
    // 处理重复键
} catch (const MySQLConnectionException& e) {  // 2. 次具体（派生类）
    // 处理连接问题
} catch (const MySQLException& e) {            // 3. 最一般（基类）
    // 处理其他数据库错误
} catch (const std::exception& e) {            // 4. 标准库基类
    // 处理所有标准异常
}

// ❌ 错误顺序：基类在前，派生类永远不会被捕获
try {
    // ...
} catch (const MySQLException& e) {            // 基类先匹配
    // 所有 MySQL 异常都会进入这里！
} catch (const MySQLDuplicateException& e) {   // ⚠️ 永远不会执行
    // ...
}
```

## 7.LOG_WARN vs LOG_ERROR 的选择
- **LOG_WARN**：业务层面的**预期情况**，不需要立即处理
- **LOG_ERROR**：系统层面的**非预期错误**，需要关注
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260202115137352.png)
```cpp
} catch (const MySQLDuplicateException& e) {
    // 业务错误：用户输入了已存在的用户名，这是预期情况
    LOG_WARN("用户已存在: {}", username);  // ✅ WARN

}
```
