# 一、Docker概述

## 1.Docker技术的用途（先有个印象即可）
在之后的章节你会学到**Dockerfile**与**docker-compose**，先明确：
- Dockerfile主要用于“**部署阶段**”，但它是学习docker-compose的基础
- docker-compose即可用于“**辅助开发**”，也用于“**部署阶段**”

### （1）三种开发模式
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260124104624779.png)
- 宿主机开发就不介绍了，主要介绍一下**Docker辅助开发**、**全Docker开发**

### （2）Docker辅助开发（推荐）
- 我们还是像日常开发一样在宿主机上写项目代码、编译项目，但**第三方服务端**通过**docker-compose**一键编排安装启动，无需在宿主机手动配置服务环境；
	- 我们日常开发时如果要使用第三方服务（比如mysql、redis等），是不是要去下载对应的第三方库，进行安装编译，最终运行“**服务端**”为我们提供服务
	- **这个过程很麻烦耗时**
- **注意**：我们**还是要去下载第三方服务的“客户端”**
	- 因为我们要用到**第三方服务客户端的“头文件”**，编译时需要
- **疑惑**：那不还是要去下载第三方库吗？
	- 第三方库的安装、编译主要是为了得到并运行第三方库的 **服务端** 程序（最耗时），但我们下载只需要 **客户端头文件** 就可以了啊
	- 第三方库的**客户端**通过命令行很容易就可以获取了
- **一句话总结**：
	- **服务端安装是"大头"，Docker 帮你搞定大头；客户端开发库本来就是"小事"，apt 一下就完了。**

### （3）全Docker开发（了解）
- 从“代码编写、编译、运行到调试”的**全流程均在Docker容器内完成**
- 宿主机仅负责存储代码、操作IDE界面，不直接参与编译环境的支撑
- 典型实现：VSCode Dev Container、JetBrains Gateway
- 宿主机需要安装什么？
	- 只需要：Docker、Docker Compose、代码编辑器（VSCode）
	- 不需要：gcc、cmake、libmysqlclient-dev 等任何开发依赖
- **一句话总结**：
	- 宿主机只当"显示器+键盘"，所有脏活累活都在容器里干。

## 2.Docker是什么？
**Docker = 把程序和它需要的环境打包在一起，走哪都能跑，用完即删不留垃圾**
Docker 是一个**开源的容器化平台**，通过**操作系统级虚拟化技术**，将应用及其依赖打包成轻量级、可移植的容器。
- Docker 容器**共享宿主机的内核**，只**隔离用户空间**（文件系统、进程、网络等），而不是虚拟化完整的操作系统。
- Docker 容器使用**基础镜像（如 Ubuntu 镜像）提供文件系统环境**，然后将程序**依赖的第三方服务（如 MySQL、Redis）以容器形式运行**，你的程序就可以在这个隔离的环境中运行了。
- **核心**：
	- 通过容器技术，为程序打包并隔离出所需的运行环境（文件系统、依赖库、配置等），使程序在这个独立的环境中运行，与宿主机和其他容器互不干扰。
	- **注意**：Docker 是**容器化**，不是虚拟化。它不虚拟化硬件或完整 OS，而是**共享宿主机内核**，只隔离用户空间

## 4.Docker技术的优势
### （1）快速获取第三方服务的“服务端”
- 在前面我们提到：
	- **第三方服务的“客户端”**：我们可以通过命令行快速获取
	- **第三方服务的“服务端”**：传统情况我们需要去下载、配置、编译、运行，这个过程繁琐费时，但**通过Docker，配置好后，一键启动即可**。
```bash
┌──────────────────────────────────────────────────────────────┐
│                    开发效率对比                               │
├────────────────────────┬─────────────────────────────────────┤
│       传统方式          │            Docker 方式              │
├────────────────────────┼─────────────────────────────────────┤
│  MySQL: 下载→安装→配置  │  docker run mysql:8.0              │
│  Redis: 下载→安装→配置  │  docker run redis:7                │
│  Kafka: 下载→Java→ZK   │  docker compose up kafka           │
├────────────────────────┼─────────────────────────────────────┤
│  耗时：数小时           │  耗时：几分钟                       │
└────────────────────────┴─────────────────────────────────────┘
```

### （2）环境隔离，不污染主机
- 通过 Docker 运行的服务都在容器内部，**不会将软件安装到宿主机上**，**删除容器即可完全清理**，不留任何残留。

### （3）提高程序可移植性
- 问题场景：
	- 我们在 Ubuntu 上开发的程序，拿到 CentOS 上运行，可能因为系统差异（库版本、路径不同等）导致运行失败
	- 依赖的第三方服务（MySQL、Redis），每换一台电脑都要重新下载安装，还可能出现版本不匹配的问题
Docker 解决方案：
	- 开发完通过**Dockerfile进行部署**，将程序和依赖**打包成镜像**
	- 在任何提供 **Linux 内核环境** 的机器上都能以相同方式运行
		- Linux 服务器：直接运行（生产环境）
		- Windows (WSL2) / macOS (Docker Desktop)：通过内置 Linux VM 运行（开发环境）
**注意：**
	- 因为Docker与所在主机**共享内核**，这也是限制Docker的因素之一

### （4）版本管理与多版本共存
- 我们可以创建多个 Docker 容器，运行不同版本的服务，互不干扰。而传统方式在同一台机器上安装多个版本的 MySQL 几乎无法实现。
```cpp
# 同一台机器同时运行 MySQL 5.7 和 MySQL 8.0
docker run -d --name mysql57 -p 3306:3306 mysql:5.7
docker run -d --name mysql80 -p 3307:3306 mysql:8.0

# 同一台机器同时运行 Redis 6 和 Redis 7
docker run -d --name redis6 -p 6379:6379 redis:6
docker run -d --name redis7 -p 6380:6379 redis:7
```

### （5）快速部署与回滚
```cpp
# 部署新版本
docker stop myapp && docker run -d myapp:v2.0

# 发现问题？秒级回滚
docker stop myapp && docker run -d myapp:v1.0
```

### （6）资源利用率高
- Docker 与虚拟机都是隔离技术，但原理完全不同：
	- **虚拟机**：需要虚拟化完整的硬件和操作系统，每个虚拟机都运行一个独立的 OS 内核
	- **Docker**：共享宿主机内核，只隔离用户空间，因此更加轻量

## 5.Docker的局限
```cpp
┌────────────────────────────────────────────────────────────┐
│  Docker 的局限：                                            │
│                                                            │
│  【内核依赖】                                                │
│  • Linux 容器只能跑 Linux 程序（共享内核的代价）             │
│  • Windows/macOS 上跑 Docker 实际是通过 Linux VM           │
│                                                            │
│  【应用场景限制】                                            │
│  • GUI 程序支持不好（主要面向服务端/后台程序）               │
│  • 对内核版本有要求的程序可能有兼容问题                      │
│                                                            │
│  【学习成本】                                                │
│  • 需要学习新的调试方式（容器内 GDB 需要特殊配置）           │
│  • 网络、存储、日志等配置有学习曲线                          │
│                                                            │
│  【其他】                                                    │
│  • 数据持久化需要额外配置（Volume）                          │
│  • 容器间通信需要理解 Docker 网络模型                        │
└────────────────────────────────────────────────────────────┘
```

```cpp
│  ✓ 统一构建环境：所有人用同一个 Docker 镜像编译                          │
│  ✓ 依赖一次配置：Dockerfile 写好后，一条命令安装所有依赖                  │
│  ✓ 环境完全一致：开发、测试、生产用同一个镜像                            │
│  ✓ 部署简单：镜像包含所有运行时依赖，走哪都能跑                          │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

## 2.Dockerfile详解（C++）
- **Dockerfile的主要用途**：
	- 在开发完项目过后，**用于项目的部署**
- 对于 **"Docker辅助开发"模式**：
	- 主要用 docker-compose 来启动第三方服务（MySQL、Redis等），Dockerfile 几乎不用
	- **但学习Dockerfile有助于学习docker-compose**
- 对于 **"全Docker开发"模式**：
	- 可能会用到 Dockerfile 来构建开发环境镜像
- **阅读顺序建议**：
	- **（2）——>（3）——>（1）**

### （1）Dockerfile的构造流程（多阶段构造）
- **Dockerfile = 镜像的"配方"**：
	- 告诉 Docker 如何一步步构建出你需要的**镜像**
- **其实就和在宿主机上构建项目、编译项目、运行项目类似，只是指令与命令行指令不太一样**

#### <1> 构建阶段（第一个 FROM）
- **目的**：
	- **在镜像中编译代码，生成可执行文件**
- **步骤**：
	- 1.在镜像中搭建依赖的系统环境（From命令）
	- 2.设置镜像中的环境变量（ENV命令）
	- 3.设置镜像中的工作目录（WORKDIR命令）
	- 4.切换到工作目录中
		- 之后的操作就在工作目录中进行了
	- 5.下载你项目所依赖的编译工具、第三方库到镜像中（RUN命令）
	- 6.将你的项目文件复制到镜像中（COPY命令）
		- 你的项目文件、依赖的系统环境、编译工具、第三方库等都到了镜像中，那么就可以在镜像中编译你的项目了
	- 7.设置在镜像中的编译、链接命令（RUN命令）
		- 如：**执行“CMake文件”** 

#### <2>运行阶段（多阶段构建的第二个 FROM，仍在 docker build 时执行）
- **目的**：
	- **运行可执行文件，生成最终的轻量镜像**
- **步骤**：
	- 1.在镜像中搭建依赖的系统环境（FROM命令）
		- “构建阶段”得到的可执行文件才可以正常运行
	- 2.下载**可执行文件**运行时需要的动态库（RUN命令）
		- 在“构建阶段”，**头文件**（自定义、第三方库等）在预编译时就通过`#include`被插入到源文件中了，所以不再需要头文件了
		- 对于**静态库**（自己生成、第三库等），静态库在链接时，就被与目标文件链接到一起生成可执行文件了
		- 综上：**只需要下载第三方库的动态库就可以了**
			- 我们自己生成的动态库已经有了
			- 对于动态库，可执行文件执行时自己去找
	- 3.从构建阶段复制可执行文件（COPY）
	- 4.设置在镜像中执行可执行文件的命令（CMD命令）

#### <3>执行流程
- **docker build**：执行 Dockerfile 中所有指令（包括两个阶段），**生成最终镜像**
- **docker run**：基于镜像创建容器，执行 CMD 中记录的命令
```cpp
┌──────────────────────────────────────────────────────────────────┐
│                    Dockerfile 工作流程                            │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│   Dockerfile          docker build           镜像 (Image)        │
│   ┌──────────┐         ─────────→         ┌──────────────┐       │
│   │ FROM ... │  ←─ 构建阶段               │ 可执行文件    │       │
│   │ RUN ...  │                            │ + 运行时依赖  │       │
│   │ FROM ... │  ←─ 运行阶段               └──────────────┘       │
│   │ CMD ...  │                                    │              │
│   └──────────┘                             docker run            │
│                                                   ↓              │
│                                            ┌──────────────┐      │
│                                            │ 容器(执行CMD) │      │
│                                            └──────────────┘      │
└──────────────────────────────────────────────────────────────────┘
```

#### <4>docker build 与 docker run 的关系
```cpp
┌─────────────────────────────────────────────────────────────────┐
│  docker build（构建镜像）                                        │
│  ───────────────────────                                        │
│  • 读取 Dockerfile，逐条执行指令                                 │
│  • 每条指令生成一个镜像层                                        │
│  • CMD 命令只是"写入"镜像，此时不执行                            │
│  • 最终输出：镜像文件                                            │
├─────────────────────────────────────────────────────────────────┤
│  docker run（运行容器）                                          │
│  ──────────────────────                                         │
│  • 基于镜像创建容器实例                                          │
│  • 执行镜像中记录的 CMD 命令                                     │
│  • 最终输出：运行中的容器                                        │
└─────────────────────────────────────────────────────────────────┘
```

#### <5>各阶段需要的内容对比
```cpp
┌────────────────────────┬───────────────────────────────────┐
│  构建阶段需要           │  运行阶段需要                      │
├────────────────────────┼───────────────────────────────────┤
│  ✓ 头文件 (.h)         │  ✗ 不需要（已预处理展开）          │
│  ✓ 静态库 (.a)         │  ✗ 不需要（已链接进可执行文件）    │
│  ✓ 编译器 (gcc)        │  ✗ 不需要                         │
│  ✓ 构建工具 (cmake)    │  ✗ 不需要                         │
│  ✓ 动态库 (.so)        │  ✓ 需要（运行时加载）              │
│  ✓ 源代码              │  ✗ 不需要                         │
│  ✗ 可执行文件          │  ✓ 需要（从构建阶段复制）          │
└────────────────────────┴───────────────────────────────────┘
```

### （2）C++通用Dockerfile模板
#### <1>详细版（带类比注释）
```cpp
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│   你在自己电脑上编译 C++ 项目的步骤：                                      │
│                                                                         │
│   1. 装一个 Ubuntu 系统                                                  │
│   2. 打开终端，apt install 安装 gcc、cmake、各种库                        │
│   3. 把项目代码放到某个目录                                               │
│   4. cd 到那个目录                                                       │
│   5. mkdir build && cd build && cmake .. && make                        │
│   6. 运行编译好的程序                                                    │
│                                                                         │
│   Dockerfile 就是把这些步骤写下来，让 Docker 自动执行                      │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```
```bash
# ╔═══════════════════════════════════════════════════════════════════════╗
# ║                        阶段一：构建阶段                                 ║
# ║     类比：你新装了一台电脑，要在上面配置编译环境、编译代码                  ║
# ╚═══════════════════════════════════════════════════════════════════════╝

# ─────────────────────────────────────────────────────────────────────────
# FROM = 我要基于什么系统来搭环境
# ─────────────────────────────────────────────────────────────────────────
# 【类比】就像你去电脑城说："给我装个 Ubuntu 22.04"
# 
# AS builder 是给这个阶段起个名字，后面会用到
FROM ubuntu:22.04 AS builder


# ─────────────────────────────────────────────────────────────────────────
# ENV = 设置环境变量
# ─────────────────────────────────────────────────────────────────────────
# 【类比】就像你在 ~/.bashrc 里 export 一个变量
# 
# DEBIAN_FRONTEND=noninteractive 的作用：
# 正常 apt install 有时会弹出蓝色界面问你 yes/no
# 设置这个变量后，就不会弹窗，直接用默认选项（Docker 构建时没人能按键盘）
ENV DEBIAN_FRONTEND=noninteractive


# ─────────────────────────────────────────────────────────────────────────
# RUN = 执行一条命令
# ─────────────────────────────────────────────────────────────────────────
# 【类比】就像你打开终端，敲命令
# 
# 下面这段等价于你在终端里敲：
#   sudo apt-get update
#   sudo apt-get install -y build-essential cmake git ...
#   sudo rm -rf /var/lib/apt/lists/*
#
# 为什么用 && 连接而不是写多个 RUN？
#   每个 RUN 会创建一个"镜像层"，层越多镜像越大
#   用 && 连接，所有命令在同一层执行，镜像更小
RUN apt-get update && apt-get install -y \
    # ┌─────────────────────────────────────────────────────────┐
    # │ 基础编译工具                                             │
    # │ 就像你编译 C++ 之前，肯定要先装 gcc 和 cmake             │
    # └─────────────────────────────────────────────────────────┘
    build-essential \
    # ↑ 这个包包含 gcc, g++, make 等，装这一个就够了
    cmake \
    git \
    pkg-config \
    # pkg-config 用来查找库的编译参数（比如 mysql 的头文件在哪）
    
    # ┌─────────────────────────────────────────────────────────┐
    # │ 项目依赖的第三方库（根据你的项目需求增删）                  │
    # │                                                         │
    # │ 注意：这里装的是 -dev 版本（开发版）                      │
    # │ -dev 版本包含头文件 .h，编译时需要                        │
    # │ 运行时只需要 .so 动态库，不需要头文件                      │
    # └─────────────────────────────────────────────────────────┘
    libmysqlclient-dev \
    # ↑ MySQL 客户端库，编译时需要 mysql.h 头文件
    libhiredis-dev \
    # ↑ Redis 客户端库（hiredis）
    libgrpc++-dev \
    libprotobuf-dev \
    protobuf-compiler \
    protobuf-compiler-grpc \
    
    # ┌─────────────────────────────────────────────────────────┐
    # │ 清理 apt 缓存，减小镜像体积                               │
    # │ /var/lib/apt/lists/* 是 apt update 下载的软件包列表      │
    # │ 装完软件就没用了，删掉能省几十 MB                          │
    # └─────────────────────────────────────────────────────────┘
    && rm -rf /var/lib/apt/lists/*


# ─────────────────────────────────────────────────────────────────────────
# WORKDIR = 切换工作目录
# ─────────────────────────────────────────────────────────────────────────
# 【类比】就像你 cd /build
#        如果目录不存在，会自动 mkdir
#        之后的命令都在这个目录下执行
WORKDIR /build


# ─────────────────────────────────────────────────────────────────────────
# COPY = 把文件从你的电脑复制到镜像里
# ─────────────────────────────────────────────────────────────────────────
# 【类比】就像你用 U 盘把代码拷到新电脑上
# 
# COPY <源路径> <目标路径>
# COPY . . 的意思：
#   第一个 . = 当前目录（你执行 docker build 的目录，即"构建上下文"）
#   第二个 . = 镜像里的当前目录（刚才 WORKDIR 设置的 /build）
# 
# 所以 COPY . . = 把项目目录下所有文件，复制到镜像的 /build 目录
COPY . .


# ─────────────────────────────────────────────────────────────────────────
# 编译项目
# ─────────────────────────────────────────────────────────────────────────
# 【类比】就像你平时编译项目：
#   mkdir build
#   cd build  
#   cmake ..
#   make -j8
#
# $(nproc) 会返回 CPU 核心数，比如 8 核就是 make -j8，并行编译更快
RUN mkdir -p build && cd build \
    && cmake .. -DCMAKE_BUILD_TYPE=Release \
    && make -j$(nproc)


# ╔═══════════════════════════════════════════════════════════════════════╗
# ║                        阶段二：运行阶段                                 ║
# ║     类比：编译完了，你要把程序拷到另一台"干净的服务器"上运行               ║
# ║           那台服务器只需要能运行程序，不需要编译工具                      ║
# ╚═══════════════════════════════════════════════════════════════════════╝
# 
# 【为什么要分两个阶段？】
# 
# 构建阶段装了一堆东西：gcc、cmake、-dev 开发库、源代码... (可能 1GB+)
# 但运行程序时，这些都不需要！只需要：
#   1. 编译好的可执行文件
#   2. 程序依赖的 .so 动态库
#
# 所以我们新开一个"干净的"阶段，只装运行必需的东西
# 这样最终镜像可能只有 100-200MB

FROM ubuntu:22.04

ENV DEBIAN_FRONTEND=noninteractive

# ─────────────────────────────────────────────────────────────────────────
# 只安装【运行时库】
# ─────────────────────────────────────────────────────────────────────────
# 【对比】
#   构建阶段装的：libmysqlclient-dev（包含头文件 + 动态库，约 50MB）
#   运行阶段装的：libmysqlclient21  （只有动态库，约 5MB）
#
# 头文件只在编译时用（#include <mysql.h>）
# 运行时程序只需要 .so 动态库
RUN apt-get update && apt-get install -y \
    libmysqlclient21 \
    libhiredis0.14 \
    libgrpc++1 \
    libprotobuf23 \
    && rm -rf /var/lib/apt/lists/*


# ─────────────────────────────────────────────────────────────────────────
# 设置运行目录
# ─────────────────────────────────────────────────────────────────────────
WORKDIR /app


# ─────────────────────────────────────────────────────────────────────────
# COPY --from=builder = 从构建阶段复制文件
# ─────────────────────────────────────────────────────────────────────────
# 【类比】编译完成后，你只把 build/my_server 这个可执行文件拷到服务器上
#        源代码、.o 文件、cmake 缓存...都不要了
#
# --from=builder 指定从哪个阶段复制（就是上面 AS builder 起的名字）
# /build/build/my_server = 构建阶段里，编译产物的路径
# . = 当前目录（/app）
COPY --from=builder /build/build/my_server .


# ─────────────────────────────────────────────────────────────────────────
# EXPOSE = 声明容器要用的端口
# ─────────────────────────────────────────────────────────────────────────
# 【注意】这只是"文档说明"，告诉使用者这个容器会监听 8080 端口
#        它不会真的开放端口，真正开放端口要用 docker run -p
EXPOSE 8080


# ─────────────────────────────────────────────────────────────────────────
# CMD = 容器启动时执行的命令
# ─────────────────────────────────────────────────────────────────────────
# 【类比】你把程序拷到服务器后，运行 ./my_server
#
# 写法：CMD ["可执行文件", "参数1", "参数2"]
# 例如：CMD ["./my_server", "--config", "/app/config.yaml"]
CMD ["./my_server"]
```

# 三、docker-compose 在C++中的使用
- **docker-compose = 多容器编排工具**，用一个 YAML 文件定义和**管理多个容器**

## 1.为什么需要多容器？
- 每个容器都运行着一个程序，我们通过 Dockerfile 构建镜像，再基于镜像创建容器来运行我们的程序
- 我们的程序用的是第三方服务（如 MySQL、Redis）的**客户端库**，必须连接第三方服务的**服务端程序**才能获得服务
- **多容器**：
	- 我们必须**为每一个依赖的第三方服务的“服务端程序”都创建一个"容器"来运行它们**

**注：**
- 对于 MySQL、Redis 这些第三方服务，不需要写 Dockerfile 来生成镜像，直接用**官方镜像**就行

## 2.docker-compose的必要性
- 虽然第三方服务有**官方镜像**，但我们要为每一个服务都手动 docker run，并需要记住各种 -p -v -e 参数
	- **每次启动要敲一堆命令，还容易敲错**
- **多个容器之间怎么互相访问？启动顺序怎么控制？**
- docker-compose 可以在一个 YAML 文件中定义所有服务，**一条命令启动全部**

## 3.C++ 通用 docker-compose.yml 模板
### （1）详细版（带注释）
```bash
# ╔═══════════════════════════════════════════════════════════════════════╗
# ║                    docker-compose.yml 详细模板                         ║
# ║     场景：C++ 服务器 + MySQL + Redis + Nginx                           ║
# ╚═══════════════════════════════════════════════════════════════════════╝

# ─────────────────────────────────────────────────────────────────────────
# version: 指定 docker-compose 文件格式版本
# ─────────────────────────────────────────────────────────────────────────
# "3.8" 是目前常用的稳定版本，支持大多数功能
version: "3.8"

# ─────────────────────────────────────────────────────────────────────────
# services: 定义所有服务（每个服务对应一个容器）
# ─────────────────────────────────────────────────────────────────────────
# 【类比】就像你在规划：我需要哪些服务器？每台服务器装什么？
services:

  # ═══════════════════════════════════════════════════════════════════════
  # 服务1：你的 C++ 服务器程序
  # ═══════════════════════════════════════════════════════════════════════
  my_server:
    # ─────────────────────────────────────────────────────────────────────
    # build: 使用 Dockerfile 构建镜像
    # ─────────────────────────────────────────────────────────────────────
    # 【类比】告诉 docker-compose：这个服务的镜像要用 Dockerfile 现场构建
    #
    # context: Dockerfile 所在目录（. 表示当前目录）
    # dockerfile: Dockerfile 文件名（默认就是 Dockerfile，可省略）
    build:
      context: .
      dockerfile: Dockerfile
    
    # ─────────────────────────────────────────────────────────────────────
    # image: 给构建的镜像起个名字
    # ─────────────────────────────────────────────────────────────────────
    image: my_server:latest
    
    # ─────────────────────────────────────────────────────────────────────
    # container_name: 容器名（方便 docker logs 查看）
    # ─────────────────────────────────────────────────────────────────────
    container_name: my_server_container
    
    # ─────────────────────────────────────────────────────────────────────
    # ports: 端口映射
    # ─────────────────────────────────────────────────────────────────────
    # 格式：宿主机端口:容器端口
    # 【类比】把容器的 8080 端口"暴露"到宿主机的 8080 端口
    #        访问 宿主机:8080 就等于访问 容器:8080
    ports:
      - "8080:8080"       # HTTP 端口
      - "50051:50051"     # gRPC 端口
    
    # ─────────────────────────────────────────────────────────────────────
    # volumes: 目录挂载（数据持久化）
    # ─────────────────────────────────────────────────────────────────────
    # 格式：宿主机路径:容器内路径
    # 【类比】把宿主机的文件夹"映射"到容器里
    #        容器删除后，宿主机上的数据还在
    #
    # 常见用途：
    #   1. 配置文件：不用重新构建镜像就能改配置
    #   2. 日志文件：方便在宿主机上查看日志
    #   3. 数据文件：数据持久化，容器删了数据还在
    volumes:
      - ./configs:/app/configs      # 配置文件
      - ./logs:/app/logs            # 日志目录
      # 只读挂载
      - ./config.yaml:/app/config.yaml:ro
    
    # ─────────────────────────────────────────────────────────────────────
    # environment: 环境变量
    # ─────────────────────────────────────────────────────────────────────
    # 【类比】就像在程序启动前 export 一些变量
    #        程序里可以用 getenv("DB_HOST") 读取
    environment:
      - DB_HOST=mysql               # 数据库地址（用服务名，不是 IP）
      - DB_PORT=3306
      - DB_USER=root
      - DB_PASSWORD=123456
      - REDIS_HOST=redis
      - REDIS_PORT=6379
    
    # ─────────────────────────────────────────────────────────────────────
    # depends_on: 依赖关系（启动顺序）
    # ─────────────────────────────────────────────────────────────────────
    # 【类比】你的 C++ 程序启动前，MySQL 和 Redis 必须先启动
    #
    # 注意：depends_on 只保证容器启动顺序
    #       不保证 MySQL 真的"准备好"了（可能还在初始化）
    #       生产环境建议程序里加重试逻辑
    #
    # 【升级】使用 condition: service_healthy 可以等待服务真正就绪
    #        需要配合目标服务的 healthcheck 使用
    depends_on:
      mysql:
        condition: service_healthy  # 等待 MySQL 健康检查通过
      redis:
        condition: service_healthy  # 等待 Redis 健康检查通过
    
    # ─────────────────────────────────────────────────────────────────────
    # networks: 加入哪个网络
    # - 把这个容器加入到名为 "app_network" 的网络中
    # - 在同一个网络中，直接用服务名，如："mysql" 作为主机名
    # - mysql_real_connect(conn, "mysql", "root", "123456", ...);
    #                               ↑ 服务名，不是 IP    	
    # ─────────────────────────────────────────────────────────────────────
    networks:
      - app_network
    
    # ─────────────────────────────────────────────────────────────────────
    # restart: 重启策略
    # ─────────────────────────────────────────────────────────────────────
    # always: 容器退出后总是重启（适合生产环境）
    # unless-stopped: 除非手动停止，否则总是重启
    # on-failure: 只有异常退出才重启
    restart: unless-stopped
    
    # ─────────────────────────────────────────────────────────────────────
    # healthcheck: 健康检查（my_server）
    # ─────────────────────────────────────────────────────────────────────
    # 【作用】定期检查容器内服务是否正常运行
    # 【原理】执行 test 命令，返回 0 表示健康，非 0 表示不健康
    #
    # test: 检查命令
    #   - 方式1：HTTP 接口检查（需要程序提供 /health 接口）
    #   - 方式2：TCP 端口检查（检查端口是否能连通）
    #   - 方式3：自定义脚本
    #
    # interval: 检查间隔（每隔多久检查一次）
    # timeout: 超时时间（命令执行超过这个时间视为失败）
    # retries: 重试次数（连续失败多少次才判定为不健康）
    # start_period: 启动宽限期（容器启动后等待多久再开始检查）
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
      # 【备选方案】如果没有 /health 接口，可以用 TCP 端口检查：
      # test: ["CMD-SHELL", "nc -z localhost 8080 || exit 1"]

  # ═══════════════════════════════════════════════════════════════════════
  # 服务2：MySQL 数据库
  # ═══════════════════════════════════════════════════════════════════════
  mysql:
    # ─────────────────────────────────────────────────────────────────────
    # image: 直接使用现成的镜像（从 Docker Hub 拉取）
    # ─────────────────────────────────────────────────────────────────────
    # 【对比】my_server 用 build 构建，mysql 用 image 拉取现成的
    image: mysql:8.0
    
    container_name: mysql_container
    
    # ─────────────────────────────────────────────────────────────────────
    # MySQL 官方镜像支持的环境变量
    # ─────────────────────────────────────────────────────────────────────
    environment:
      - MYSQL_ROOT_PASSWORD=123456    # root 密码（必须设置）
      - MYSQL_DATABASE=mydb           # 自动创建的数据库名
      - MYSQL_USER=myuser             # 自动创建的用户
      - MYSQL_PASSWORD=mypassword     # 该用户的密码
    
    ports:
      - "3307:3306" # 宿主机 3307 → 容器 3306： 避免与宿主机上3306运行的mysql发送"端口冲突"
    
    # ─────────────────────────────────────────────────────────────────────
    # 数据持久化：MySQL 数据存到宿主机
    # ─────────────────────────────────────────────────────────────────────
    # /var/lib/mysql 是 MySQL 容器内存放数据的目录
    # /docker-entrypoint-initdb.d 是"初始化脚本目录"：MySQL 容器【首次启动】时，会自动执行这个目录下的脚本
    #   - .sql 文件 → 自动用 mysql 命令执行  .sh  文件 → 自动用 bash 执行  .sql.gz   → 自动解压后执行    
    # 挂载到宿主机后，即使容器删除，数据也不会丢
    volumes:
      # 方式1：命名卷（Docker 管理，你不用管存哪）
      - mysql_data:/var/lib/mysql

      # 方式2：绑定挂载（你指定宿主机路径）——>有 "./"，这是宿主机当前目录下的 mysql_data 文件夹
      # - 宿主机路径 : 容器内路径 
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql  # 初始化脚本
    
    networks:
      - app_network
    
    restart: unless-stopped
    
    # ─────────────────────────────────────────────────────────────────────
    # healthcheck: 健康检查（MySQL）
    # ─────────────────────────────────────────────────────────────────────
    # 【作用】检查 MySQL 是否真正可以接受连接
    # 【为什么需要】MySQL 容器启动 ≠ MySQL 服务就绪
    #              MySQL 启动后还需要初始化，可能需要 10-30 秒
    #
    # mysqladmin ping: MySQL 自带的健康检查命令
    #   -h localhost: 连接本地
    #   -u root: 使用 root 用户
    #   -p$$MYSQL_ROOT_PASSWORD: 使用环境变量中的密码
    #                            $$ 是 YAML 中转义 $，实际传递 $MYSQL_ROOT_PASSWORD
    #
    # start_period: 30s 是因为 MySQL 首次启动初始化比较慢
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-p$$MYSQL_ROOT_PASSWORD"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
      # 【备选方案】更严格的检查（能执行 SQL）：
      # test: ["CMD", "mysql", "-h", "localhost", "-u", "root", "-p$$MYSQL_ROOT_PASSWORD", "-e", "SELECT 1"]

  # ═══════════════════════════════════════════════════════════════════════
  # 服务3：Redis 缓存
  # ═══════════════════════════════════════════════════════════════════════
  redis:
    image: redis:7-alpine       # alpine 版本更小
    
    container_name: redis_container
    
    ports:
      - "6379:6379"
    
    # ─────────────────────────────────────────────────────────────────────
    # command: 覆盖默认启动命令
    # ─────────────────────────────────────────────────────────────────────
    # 添加密码认证
    command: redis-server --requirepass 123456
    
    volumes:
      - redis_data:/data
    
    networks:
      - app_network
    
    restart: unless-stopped
    
    # ─────────────────────────────────────────────────────────────────────
    # healthcheck: 健康检查（Redis）
    # ─────────────────────────────────────────────────────────────────────
    # 【作用】检查 Redis 是否可以正常响应命令
    #
    # redis-cli ping: Redis 自带的健康检查命令
    #   -a 123456: 密码认证（因为上面设置了 requirepass）
    #
    # 【返回值】
    #   - 成功返回 "PONG"，退出码 0
    #   - 失败返回错误信息，退出码非 0
    #
    # Redis 启动很快，所以 start_period 可以短一些
    healthcheck:
      test: ["CMD", "redis-cli", "-a", "123456", "ping"]
      interval: 10s
      timeout: 5s
      retries: 3
      start_period: 5s
      # 【备选方案】无密码的 Redis：
      # test: ["CMD", "redis-cli", "ping"]

  # ═══════════════════════════════════════════════════════════════════════
  # 服务4：Nginx 反向代理（可选）
  # ═══════════════════════════════════════════════════════════════════════
  nginx:
    image: nginx:alpine
    
    container_name: nginx_container
    
    ports:
      - "80:80"
      - "443:443"
    
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf      # Nginx 配置
      - ./certs:/etc/nginx/certs                # SSL 证书
    
    depends_on:
      my_server:
        condition: service_healthy  # 等待 my_server 健康检查通过
    
    networks:
      - app_network
    
    restart: unless-stopped
    
    # ─────────────────────────────────────────────────────────────────────
    # healthcheck: 健康检查（Nginx）
    # ─────────────────────────────────────────────────────────────────────
    # 【作用】检查 Nginx 是否正常运行并能响应请求
    #
    # curl -f: 如果 HTTP 状态码不是 2xx/3xx 则失败
    # http://localhost/health: 需要在 nginx.conf 中配置 /health 路由
    #
    # 【简单方案】直接检查首页
    # 【生产方案】配置专门的 /health 端点
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 10s
      # 【备选方案】检查 Nginx 进程是否存在：
      # test: ["CMD-SHELL", "nginx -t && kill -0 $(cat /var/run/nginx.pid)"]
      # 【备选方案】简单检查首页：
      # test: ["CMD", "curl", "-f", "http://localhost/"]

# ─────────────────────────────────────────────────────────────────────────
# networks: 定义网络
# ─────────────────────────────────────────────────────────────────────────
# 【类比】把所有服务放在同一个"局域网"里
#        服务之间可以用服务名互相访问（如 mysql:3306）
#        不需要知道对方的 IP 地址
networks:
  app_network:
    driver: bridge              # 桥接网络（默认）

# ─────────────────────────────────────────────────────────────────────────
# volumes: 定义命名卷（数据持久化）
# ─────────────────────────────────────────────────────────────────────────
# 【类比】创建"数据仓库"，由 Docker 管理
#        比直接写宿主机路径更方便（不用关心具体存在哪）
volumes:
  mysql_data:                   # MySQL 数据
  redis_data:                   # Redis 数据

```

```bash
# ╔═══════════════════════════════════════════════════════════════════════╗

# ─────────────────────────────────────────────────────────────────────────
# services: 定义所有服务（每个服务对应一个容器）
# ─────────────────────────────────────────────────────────────────────────
# 【类比】就像你在规划：我需要哪些服务器？每台服务器装什么？
services:

  # ═══════════════════════════════════════════════════════════════════════

    # ─────────────────────────────────────────────────────────────────────
    # build: 使用 Dockerfile 构建镜像
    # ─────────────────────────────────────────────────────────────────────
    # 【类比】告诉 docker-compose：这个服务的镜像要用 Dockerfile 现场构建
    #
    # context: Dockerfile 所在目录（. 表示当前目录）
    # dockerfile: Dockerfile 文件名（默认就是 Dockerfile，可省略）
    build:
      context: .
      dockerfile: Dockerfile
    
    # ─────────────────────────────────────────────────────────────────────
    # image: 给构建的镜像起个名字
    # ─────────────────────────────────────────────────────────────────────
    image: my_server:latest
    
    # ─────────────────────────────────────────────────────────────────────
    # container_name: 容器名（方便 docker logs 查看）
    # ─────────────────────────────────────────────────────────────────────
    container_name: my_server_container
    
    # ─────────────────────────────────────────────────────────────────────
    # ports: 端口映射
    # ─────────────────────────────────────────────────────────────────────
    # 格式：宿主机端口:容器端口
    # 【类比】把容器的 8080 端口"暴露"到宿主机的 8080 端口
    #        访问 宿主机:8080 就等于访问 容器:8080
    ports:
      - "8080:8080"       # HTTP 端口
      - "50051:50051"     # gRPC 端口
    
    # ─────────────────────────────────────────────────────────────────────
    # volumes: 目录挂载（数据持久化）
    # ─────────────────────────────────────────────────────────────────────
    # 格式：宿主机路径:容器内路径
    # 【类比】把宿主机的文件夹"映射"到容器里
    #        容器删除后，宿主机上的数据还在
    #
    # 常见用途：
    #   1. 配置文件：不用重新构建镜像就能改配置
    #   2. 日志文件：方便在宿主机上查看日志
    #   3. 数据文件：数据持久化，容器删了数据还在
    volumes:
      - ./configs:/app/configs      # 配置文件
      - ./logs:/app/logs            # 日志目录
      # 只读挂载
      - ./config.yaml:/app/config.yaml:ro
    
    # ─────────────────────────────────────────────────────────────────────
    # environment: 环境变量
    # ─────────────────────────────────────────────────────────────────────
    # 【类比】就像在程序启动前 export 一些变量
    #        程序里可以用 getenv("DB_HOST") 读取
    environment:
      - DB_HOST=mysql               # 数据库地址（用服务名，不是 IP）
      - DB_PORT=3306
      - DB_USER=root
      - DB_PASSWORD=123456
      - REDIS_HOST=redis
      - REDIS_PORT=6379
      - KAFKA_BROKERS=kafka:9092    # Kafka 地址（容器内部通信用 9092）
    
    # ─────────────────────────────────────────────────────────────────────
    # depends_on: 依赖关系（启动顺序）
    # ─────────────────────────────────────────────────────────────────────
    # 【类比】你的 C++ 程序启动前，MySQL 和 Redis 必须先启动
    #
    # 注意：depends_on 只保证容器启动顺序
    #       不保证 MySQL 真的"准备好"了（可能还在初始化）
    #       生产环境建议程序里加重试逻辑
    #
    # 【升级】使用 condition: service_healthy 可以等待服务真正就绪
    #        需要配合目标服务的 healthcheck 使用
    depends_on:
      mysql:
        condition: service_healthy  # 等待 MySQL 健康检查通过
      redis:
        condition: service_healthy  # 等待 Redis 健康检查通过
      kafka:
        condition: service_healthy  # 等待 Kafka 健康检查通过
    
    # ─────────────────────────────────────────────────────────────────────
    # networks: 加入哪个网络
    # - 把这个容器加入到名为 "app_network" 的网络中
    # - 在同一个网络中，直接用服务名，如："mysql" 作为主机名
    # - mysql_real_connect(conn, "mysql", "root", "123456", ...);     
    #                               ↑ 服务名，不是 IP    	
    # ─────────────────────────────────────────────────────────────────────
    networks:
      - app_network
    
    # ─────────────────────────────────────────────────────────────────────
    # restart: 重启策略
    # ─────────────────────────────────────────────────────────────────────
    # always: 容器退出后总是重启（适合生产环境）
    # unless-stopped: 除非手动停止，否则总是重启
    # on-failure: 只有异常退出才重启
    restart: unless-stopped
    
    # ─────────────────────────────────────────────────────────────────────
    # healthcheck: 健康检查（my_server）
    # ─────────────────────────────────────────────────────────────────────
    # 【作用】定期检查容器内服务是否正常运行
    # 【原理】执行 test 命令，返回 0 表示健康，非 0 表示不健康
    #
    # test: 检查命令
    #   - 方式1：HTTP 接口检查（需要程序提供 /health 接口）
    #   - 方式2：TCP 端口检查（检查端口是否能连通）
    #   - 方式3：自定义脚本
    #
    # interval: 检查间隔（每隔多久检查一次）
    # timeout: 超时时间（命令执行超过这个时间视为失败）
    # retries: 重试次数（连续失败多少次才判定为不健康）
    # start_period: 启动宽限期（容器启动后等待多久再开始检查）
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
      # 【备选方案】如果没有 /health 接口，可以用 TCP 端口检查：
      # test: ["CMD-SHELL", "nc -z localhost 8080 || exit 1"]

  # ═══════════════════════════════════════════════════════════════════════
  # 服务2：MySQL 数据库
  # ═══════════════════════════════════════════════════════════════════════
  mysql:
    # ─────────────────────────────────────────────────────────────────────
    # image: 直接使用现成的镜像（从 Docker Hub 拉取）
    # ─────────────────────────────────────────────────────────────────────
    # 【对比】my_server 用 build 构建，mysql 用 image 拉取现成的
    image: mysql:8.0
    
    container_name: mysql_container
    
    # ─────────────────────────────────────────────────────────────────────
    # MySQL 官方镜像支持的环境变量
    # ─────────────────────────────────────────────────────────────────────
    environment:
      - MYSQL_ROOT_PASSWORD=123456    # root 密码（必须设置）
      - MYSQL_DATABASE=mydb           # 自动创建的数据库名
      - MYSQL_USER=myuser             # 自动创建的用户
      - MYSQL_PASSWORD=mypassword     # 该用户的密码
    
    ports:
      - "3307:3306" # 宿主机 3307 → 容器 3306： 避免与宿主机上3306运行的mysql发送"端口冲突"
    
    # ─────────────────────────────────────────────────────────────────────
    # 数据持久化：MySQL 数据存到宿主机
    # ─────────────────────────────────────────────────────────────────────
    # /var/lib/mysql 是 MySQL 容器内存放数据的目录
    # /docker-entrypoint-initdb.d 是"初始化脚本目录"：MySQL 容器【首次启动】时，会自动执行这个目录下的脚本
    #   - .sql 文件 → 自动用 mysql 命令执行  .sh  文件 → 自动用 bash 执行  .sql.gz   → 自动解压后执行    
    # 挂载到宿主机后，即使容器删除，数据也不会丢
    volumes:
      # 方式1：命名卷（Docker 管理，你不用管存哪）
      - mysql_data:/var/lib/mysql

      # 方式2：绑定挂载（你指定宿主机路径）——>有 "./"，这是宿主机当前目录下的 mysql_data 文件夹
      # - 宿主机路径 : 容器内路径 
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql  # 初始化脚本
    
    networks:
      - app_network
    
    restart: unless-stopped
    
    # ─────────────────────────────────────────────────────────────────────
    # healthcheck: 健康检查（MySQL）
    # ─────────────────────────────────────────────────────────────────────
    # 【作用】检查 MySQL 是否真正可以接受连接
    # 【为什么需要】MySQL 容器启动 ≠ MySQL 服务就绪
    #              MySQL 启动后还需要初始化，可能需要 10-30 秒
    #
    # mysqladmin ping: MySQL 自带的健康检查命令
    #   -h localhost: 连接本地
    #   -u root: 使用 root 用户
    #   -p$$MYSQL_ROOT_PASSWORD: 使用环境变量中的密码
    #                            $$ 是 YAML 中转义 $，实际传递 $MYSQL_ROOT_PASSWORD
    #
    # start_period: 30s 是因为 MySQL 首次启动初始化比较慢
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-p$$MYSQL_ROOT_PASSWORD"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
      # 【备选方案】更严格的检查（能执行 SQL）：
      # test: ["CMD", "mysql", "-h", "localhost", "-u", "root", "-p$$MYSQL_ROOT_PASSWORD", "-e", "SELECT 1"]

  # ═══════════════════════════════════════════════════════════════════════
  # 服务3：Redis 缓存
  # ═══════════════════════════════════════════════════════════════════════
  redis:
    image: redis:7-alpine       # alpine 版本更小
    
    container_name: redis_container
    
    ports:
      - "6379:6379"
    
    # ─────────────────────────────────────────────────────────────────────
    # command: 覆盖默认启动命令
    # ─────────────────────────────────────────────────────────────────────
    # 添加密码认证
    command: redis-server --requirepass 123456
    
    volumes:
      - redis_data:/data
    
    networks:
      - app_network
    
    restart: unless-stopped
    
    # ─────────────────────────────────────────────────────────────────────
    # healthcheck: 健康检查（Redis）
    # ─────────────────────────────────────────────────────────────────────
    # 【作用】检查 Redis 是否可以正常响应命令
    #
    # redis-cli ping: Redis 自带的健康检查命令
    #   -a 123456: 密码认证（因为上面设置了 requirepass）
    #
    # 【返回值】
    #   - 成功返回 "PONG"，退出码 0
    #   - 失败返回错误信息，退出码非 0
    #
    # Redis 启动很快，所以 start_period 可以短一些
    healthcheck:
      test: ["CMD", "redis-cli", "-a", "123456", "ping"]
      interval: 10s
      timeout: 5s
      retries: 3
      start_period: 5s
      # 【备选方案】无密码的 Redis：
      # test: ["CMD", "redis-cli", "ping"]

  # ═══════════════════════════════════════════════════════════════════════
  # 服务4：Zookeeper（Kafka 依赖）
  # ═══════════════════════════════════════════════════════════════════════
  zookeeper:
    # ─────────────────────────────────────────────────────────────────────
    # image: Confluent 官方 Zookeeper 镜像
    # ─────────────────────────────────────────────────────────────────────
    # 【为什么需要 Zookeeper】
    #   Kafka 使用 Zookeeper 来：
    #   1. 管理 Broker 集群元数据
    #   2. 选举 Controller
    #   3. 存储 Topic 配置信息
    #   4. 管理消费者组（旧版本）
    #
    # 【镜像选择】
    #   - confluentinc/cp-zookeeper: Confluent 官方，功能完整，推荐
    #   - bitnami/zookeeper: Bitnami 维护，也很流行
    #   - zookeeper: Apache 官方，配置稍复杂
    image: confluentinc/cp-zookeeper:7.5.0
    
    container_name: zookeeper_container
    
    # ─────────────────────────────────────────────────────────────────────
    # Zookeeper 环境变量
    # ─────────────────────────────────────────────────────────────────────
    # ZOOKEEPER_CLIENT_PORT: 客户端连接端口（Kafka 连接用）
    # ZOOKEEPER_TICK_TIME: 心跳间隔（毫秒），用于会话超时计算
    #                      最小会话超时 = 2 * tickTime = 4000ms
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181
      ZOOKEEPER_TICK_TIME: 2000
      # 【可选配置】
      # ZOOKEEPER_INIT_LIMIT: 5     # Follower 初始化连接到 Leader 的超时（tickTime 的倍数）
      # ZOOKEEPER_SYNC_LIMIT: 2     # Follower 与 Leader 同步的超时（tickTime 的倍数）
    
    ports:
      # ─────────────────────────────────────────────────────────────────────
      # Zookeeper 端口说明
      # ─────────────────────────────────────────────────────────────────────
      # 2181: 客户端连接端口（Kafka、命令行工具连接用）
      # 2888: Follower 连接 Leader 端口（集群内部通信）
      # 3888: Leader 选举端口（集群内部通信）
      #
      # 单机部署只需要暴露 2181
      - "2181:2181"
    
    volumes:
      # ─────────────────────────────────────────────────────────────────────
      # Zookeeper 数据持久化
      # ─────────────────────────────────────────────────────────────────────
      # /var/lib/zookeeper/data: 存储快照数据
      # /var/lib/zookeeper/log: 存储事务日志
      #
      # 【重要】生产环境必须持久化，否则重启后 Kafka 元数据丢失
      - zookeeper_data:/var/lib/zookeeper/data
      - zookeeper_log:/var/lib/zookeeper/log
    
    networks:
      - app_network
    
    restart: unless-stopped
    
    # ─────────────────────────────────────────────────────────────────────
    # healthcheck: 健康检查（Zookeeper）
    # ─────────────────────────────────────────────────────────────────────
    # 【作用】检查 Zookeeper 是否正常运行
    #
    # 方法1: 使用 nc 发送 "ruok"（are you ok）命令
    #        Zookeeper 回复 "imok" 表示正常
    #
    # 方法2: 使用 zkServer.sh status 检查状态
    #
    # 【Zookeeper 四字命令】
    #   ruok: 检查服务是否运行
    #   stat: 查看服务状态和连接信息
    #   conf: 查看配置信息
    #   mntr: 查看监控指标
    healthcheck:
      test: ["CMD-SHELL", "echo ruok | nc localhost 2181 | grep imok"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 15s
      # 【备选方案】如果 nc 不可用：
      # test: ["CMD-SHELL", "zkServer.sh status"]

  # ═══════════════════════════════════════════════════════════════════════
  # 服务5：Kafka 消息队列
  # ═══════════════════════════════════════════════════════════════════════
  kafka:
    # ─────────────────────────────────────────────────────────────────────
    # image: Confluent 官方 Kafka 镜像
    # ─────────────────────────────────────────────────────────────────────
    # 【镜像选择】
    #   - confluentinc/cp-kafka: Confluent 官方，功能完整，推荐
    #   - bitnami/kafka: Bitnami 维护，配置简单
    #   - apache/kafka: Apache 官方（3.7+ 支持 KRaft 无 Zookeeper 模式）
    #
    # 【版本对应】
    #   cp-kafka:7.5.0 对应 Kafka 3.5.x
    image: confluentinc/cp-kafka:7.5.0
    
    container_name: kafka_container
    
    # ─────────────────────────────────────────────────────────────────────
    # depends_on: Kafka 依赖 Zookeeper
    # ─────────────────────────────────────────────────────────────────────
    # Kafka 启动时需要连接 Zookeeper 获取集群元数据
    # 必须等 Zookeeper 就绪后才能启动 Kafka
    depends_on:
      zookeeper:
        condition: service_healthy
    
    # ─────────────────────────────────────────────────────────────────────
    # Kafka 环境变量（重要！）
    # ─────────────────────────────────────────────────────────────────────
    environment:
      # ─────────────────────────────────────────────────────────────────
      # KAFKA_BROKER_ID: Broker 唯一标识
      # ─────────────────────────────────────────────────────────────────
      # 集群中每个 Broker 必须有唯一 ID
      # 单机部署设为 1 即可
      KAFKA_BROKER_ID: 1
      
      # ─────────────────────────────────────────────────────────────────
      # KAFKA_ZOOKEEPER_CONNECT: Zookeeper 连接地址
      # ─────────────────────────────────────────────────────────────────
      # 格式：host:port[,host:port...]
      # 使用 Docker 服务名 "zookeeper"，不是 IP
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
      
      # ─────────────────────────────────────────────────────────────────
      # KAFKA_LISTENERS: Kafka 监听的地址（Broker 内部）
      # ─────────────────────────────────────────────────────────────────
      # 定义 Broker 在哪些地址/端口监听
      # 格式：LISTENER_NAME://host:port
      #
      # PLAINTEXT: 容器内部通信（其他容器访问 kafka:9092）
      # PLAINTEXT_HOST: 宿主机访问（localhost:29092）
      KAFKA_LISTENERS: PLAINTEXT://0.0.0.0:9092,PLAINTEXT_HOST://0.0.0.0:29092
      
      # ─────────────────────────────────────────────────────────────────
      # KAFKA_ADVERTISED_LISTENERS: 对外公布的地址（客户端使用）
      # ─────────────────────────────────────────────────────────────────
      # 【关键配置】客户端连接 Kafka 后，Kafka 返回这个地址给客户端
      #
      # PLAINTEXT://kafka:9092
      #   - 容器内其他服务使用（如 my_server）
      #   - 使用服务名 "kafka"
      #
      # PLAINTEXT_HOST://localhost:29092
      #   - 宿主机上的程序使用（如本地调试）
      #   - 使用 localhost
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092,PLAINTEXT_HOST://localhost:29092
      
      # ─────────────────────────────────────────────────────────────────
      # KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: 监听器协议映射
      # ─────────────────────────────────────────────────────────────────
      # 定义每个 Listener 使用的安全协议
      # PLAINTEXT = 无加密无认证（开发环境）
      # SSL = TLS 加密
      # SASL_PLAINTEXT = SASL 认证无加密
      # SASL_SSL = SASL 认证 + TLS 加密
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: PLAINTEXT:PLAINTEXT,PLAINTEXT_HOST:PLAINTEXT
      
      # ─────────────────────────────────────────────────────────────────
      # KAFKA_INTER_BROKER_LISTENER_NAME: Broker 间通信使用的 Listener
      # ─────────────────────────────────────────────────────────────────
      KAFKA_INTER_BROKER_LISTENER_NAME: PLAINTEXT
      
      # ─────────────────────────────────────────────────────────────────
      # KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 内部 Topic 副本数
      # ─────────────────────────────────────────────────────────────────
      # __consumer_offsets Topic 的副本数
      # 单机部署必须设为 1（因为只有 1 个 Broker）
      # 生产环境建议 3（至少 3 个 Broker）
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1
      
      # ─────────────────────────────────────────────────────────────────
      # 【可选配置】
      # ─────────────────────────────────────────────────────────────────
      # KAFKA_AUTO_CREATE_TOPICS_ENABLE: true     # 自动创建 Topic（生产环境建议 false）
      # KAFKA_NUM_PARTITIONS: 3                   # 默认分区数
      # KAFKA_DEFAULT_REPLICATION_FACTOR: 1       # 默认副本数
      # KAFKA_LOG_RETENTION_HOURS: 168            # 日志保留时间（小时）
      # KAFKA_LOG_RETENTION_BYTES: 1073741824     # 日志保留大小（字节）
      KAFKA_AUTO_CREATE_TOPICS_ENABLE: "true"
      KAFKA_NUM_PARTITIONS: 3
    
    ports:
      # ─────────────────────────────────────────────────────────────────────
      # Kafka 端口说明
      # ─────────────────────────────────────────────────────────────────────
      # 9092: 容器内部通信端口（其他容器用 kafka:9092 访问）
      # 29092: 宿主机访问端口（本地调试用 localhost:29092）
      #
      # 【为什么要两个端口？】
      #   Docker 容器内和宿主机是不同的网络环境
      #   容器内用服务名 "kafka"，宿主机用 "localhost"
      #   需要不同的 Listener 来处理
      - "9092:9092"       # 容器内通信（可选暴露，调试用）
      - "29092:29092"     # 宿主机访问
    
    volumes:
      # ─────────────────────────────────────────────────────────────────────
      # Kafka 数据持久化
      # ─────────────────────────────────────────────────────────────────────
      # /var/lib/kafka/data: Kafka 日志数据存储目录
      #
      # 【重要】生产环境必须持久化，否则重启后消息丢失
      - kafka_data:/var/lib/kafka/data
    
    networks:
      - app_network
    
    restart: unless-stopped
    
    # ─────────────────────────────────────────────────────────────────────
    # healthcheck: 健康检查（Kafka）
    # ─────────────────────────────────────────────────────────────────────
    # 【作用】检查 Kafka Broker 是否正常运行
    #
    # 方法1: 使用 kafka-broker-api-versions 命令
    #        这个命令会连接 Broker 并获取 API 版本，成功即表示 Broker 正常
    #
    # 方法2: 使用 nc 检查端口（简单但不够准确）
    #
    # 方法3: 使用 kafka-metadata（KRaft 模式）
    #
    # --bootstrap-server: 要检查的 Broker 地址
    healthcheck:
      test: ["CMD-SHELL", "kafka-broker-api-versions --bootstrap-server localhost:9092"]
      interval: 10s
      timeout: 10s
      retries: 5
      start_period: 30s
      # 【备选方案】简单端口检查：
      # test: ["CMD-SHELL", "nc -z localhost 9092"]
      # 【备选方案】检查 Topic 列表：
      # test: ["CMD-SHELL", "kafka-topics --bootstrap-server localhost:9092 --list"]

  # ═══════════════════════════════════════════════════════════════════════
  # 服务6：Nginx 反向代理（可选）

  nginx:
    image: nginx:alpine
    
    container_name: nginx_container
    
    ports:
      - "80:80"
      - "443:443"
    
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf      # Nginx 配置
      - ./certs:/etc/nginx/certs                # SSL 证书
    
    depends_on:
      my_server:
        condition: service_healthy  # 等待 my_server 健康检查通过
    
    networks:
      - app_network
    
    restart: unless-stopped
    
    # ─────────────────────────────────────────────────────────────────────
    # healthcheck: 健康检查（Nginx）
    # ─────────────────────────────────────────────────────────────────────
    # 【作用】检查 Nginx 是否正常运行并能响应请求
    #
    # curl -f: 如果 HTTP 状态码不是 2xx/3xx 则失败
    # http://localhost/health: 需要在 nginx.conf 中配置 /health 路由
    #
    # 【简单方案】直接检查首页
    # 【生产方案】配置专门的 /health 端点
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 10s
      # 【备选方案】检查 Nginx 进程是否存在：
      # test: ["CMD-SHELL", "nginx -t && kill -0 $(cat /var/run/nginx.pid)"]
      # 【备选方案】简单检查首页：
      # test: ["CMD", "curl", "-f", "http://localhost/"]

# ─────────────────────────────────────────────────────────────────────────
# networks: 定义网络
# ─────────────────────────────────────────────────────────────────────────
# 【类比】把所有服务放在同一个"局域网"里
#        服务之间可以用服务名互相访问（如 mysql:3306）
#        不需要知道对方的 IP 地址
networks:
  app_network:
    driver: bridge              # 桥接网络（默认）

# ─────────────────────────────────────────────────────────────────────────
# volumes: 定义命名卷（数据持久化）
# ─────────────────────────────────────────────────────────────────────────
# 【类比】创建"数据仓库"，由 Docker 管理
#        比直接写宿主机路径更方便（不用关心具体存在哪）
volumes:
  mysql_data:                   # MySQL 数据
  redis_data:                   # Redis 数据
  zookeeper_data:               # Zookeeper 快照数据
  zookeeper_log:                # Zookeeper 事务日志
  kafka_data:                   # Kafka 消息日志
```

### （2）ports - 端口映射
- 需要注意端口冲突问题：如果宿主机上的mysql在用3306端口，则就不可以将宿主机3306端口映射到容器端口了
```bash
ports:
  - "8080:8080"           # 宿主机8080 → 容器8080
  - "3000:80"             # 宿主机3000 → 容器80
  - "127.0.0.1:3306:3306" # 只允许本机访问
```

### （6）networks - 网络配置
```bash
# networks：定义服务所属网络（网络隔离）
services:
  my_server:
    networks:
      - frontend    # 加入 frontend 网络（对外，给 Nginx 访问）
      - backend     # 加入 backend 网络（对内，访问数据库）
      #
      # my_server 同时在两个网络，可以：
      #   - 被 frontend 网络的 nginx 访问
      #   - 访问 backend 网络的 mysql
  
  mysql:
    networks:
      - backend     # 只在 backend 网络
      #
      # mysql 只在 backend，意味着：
      #   ✓ my_server 能访问它（都在 backend）
      #   ✗ nginx 不能直接访问它（nginx 只在 frontend）
      #   → 数据库被"隔离"保护起来了

networks:
  frontend:         # 前端网络：nginx ↔ my_server
  backend:          # 后端网络：my_server ↔ mysql/redis
```

### （7）healthcheck - 健康检查
```bash
# healthcheck：定义容器健康检查策略
mysql:
  image: mysql:8.0
  healthcheck:
    # test: 健康检查命令
    #   - 返回 0 = 健康
    #   - 返回非0 = 不健康
    test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
    
    interval: 10s       # 每隔多久检查一次
    timeout: 5s         # 单次检查超时时间
    retries: 5          # 连续失败几次才判定为 unhealthy
    start_period: 30s   # 容器启动后的"宽限期"，这段时间内检查失败不算数
```
```cpp
┌─────────────────────────────────────────────────────────────────┐
│                   healthcheck 参数详解                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  test         检查命令，有三种写法：                              │
│               - ["CMD", "命令", "参数"]     # 直接执行            │
│               - ["CMD-SHELL", "shell命令"] # 用 /bin/sh -c 执行  │
│               - "shell命令"                # 简写，同上           │
│                                                                 │
│  interval     检查间隔，默认 30s                                 │
│                                                                 │
│  timeout      单次检查超时，默认 30s                             │
│                                                                 │
│  retries      连续失败次数阈值，默认 3                           │
│                                                                 │
│  start_period 启动宽限期，默认 0s                                │
│               MySQL/ES 等服务启动慢，建议设置 30s~60s            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
#### <1> 常见服务的 healthcheck 写法
```bash
services:
  # ─────────────────────────────────────────────────────────────────
  # MySQL 健康检查
  # ─────────────────────────────────────────────────────────────────
  mysql:
    image: mysql:8.0
    environment:
      - MYSQL_ROOT_PASSWORD=123456
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-uroot", "-p$$MYSQL_ROOT_PASSWORD"]
      #                                                       ↑ $$ 转义，实际传 $MYSQL_ROOT_PASSWORD
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

  # ─────────────────────────────────────────────────────────────────
  # Redis 健康检查
  # ─────────────────────────────────────────────────────────────────
  redis:
    image: redis:7-alpine
    command: redis-server --requirepass 123456
    healthcheck:
      test: ["CMD", "redis-cli", "-a", "123456", "ping"]
      # 无密码版：test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 3
      start_period: 5s

  # ─────────────────────────────────────────────────────────────────
  # 自己的 C++ 服务健康检查（需要程序提供 /health 接口）
  # ─────────────────────────────────────────────────────────────────
  my_server:
    build: .
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      # 或用 wget：test: ["CMD", "wget", "-q", "--spider", "http://localhost:8080/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s

  # ─────────────────────────────────────────────────────────────────
  # Nginx 健康检查
  # ─────────────────────────────────────────────────────────────────
  nginx:
    image: nginx:alpine
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost/"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 10s
```

# 四、 .dockerignore 文件在C++中的使用
## 1.什么是 .dockerignore
```cpp
┌─────────────────────────────────────────────────────────────────────────┐
│                    .dockerignore 是什么？                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   【类比】就像 .gitignore 告诉 Git 忽略哪些文件                           │
│          .dockerignore 告诉 Docker 构建镜像时忽略哪些文件                 │
│                                                                         │
│   【作用】                                                               │
│     1. 减小构建上下文大小 → 加快 docker build 速度                        │
│     2. 避免敏感文件被复制到镜像中（如 .env、私钥）                         │
│     3. 避免不必要的缓存失效 → 利用 Docker 层缓存                          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 2.工作原理
```bash
┌─────────────────────────────────────────────────────────────────────────┐
│                    构建上下文 (Build Context)                            │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   执行 docker build 时：                                                 │
│                                                                         │
│   $ docker build -t myapp .                                             │
│                          ↑                                              │
│                    这个 "." 就是构建上下文                                │
│                                                                         │
│   Docker 会把构建上下文中的【所有文件】打包发送给 Docker 引擎              │
│                                                                         │
│   ┌─────────────────┐        发送整个目录        ┌─────────────────┐    │
│   │   你的项目目录   │  ───────────────────────→  │  Docker 引擎    │    │
│   │  (构建上下文)    │       可能很大！           │  (执行构建)     │    │
│   └─────────────────┘                            └─────────────────┘    │
│                                                                         │
│   【问题】如果目录下有 build/、.git/、node_modules/ 等大目录              │
│          发送时间会很长，而且这些文件根本不需要！                          │
│                                                                         │
│   【解决】.dockerignore 指定哪些文件不要发送                              │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 3.基本语法
```bash
# ─────────────────────────────────────────────────────────────────────────
# .dockerignore 语法（与 .gitignore 类似）
# ─────────────────────────────────────────────────────────────────────────

# 注释以 # 开头

# 忽略具体文件
README.md
LICENSE

# 忽略具体目录（目录名后加 / 更明确）
build/
cmake-build-*/
.git/

# 通配符 * 匹配任意字符（不含路径分隔符）
*.log
*.tmp
*.swp

# 通配符 ** 匹配任意路径（包含子目录）
**/*.log          # 忽略所有目录下的 .log 文件
**/temp/          # 忽略所有名为 temp 的目录

# 通配符 ? 匹配单个字符
file?.txt         # 匹配 file1.txt, fileA.txt 等

# 取反 ! 表示不忽略（例外）
*.md
!README.md        # 忽略所有 .md，但保留 README.md

# 忽略根目录下的特定文件（以 / 开头）
/config.local.yaml   # 只忽略根目录的，不忽略子目录的
```

## 5.常见问题
### （1）为什么忽略 Dockerfile？
```bash
┌─────────────────────────────────────────────────────────────────────────┐
│  Q: Dockerfile 不是构建必需的吗？为什么要忽略？                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  A: docker build 命令会单独读取 Dockerfile，不需要通过构建上下文传递      │
│                                                                         │
│     docker build -f Dockerfile -t myapp .                               │
│                  ↑                      ↑                               │
│            单独指定 Dockerfile      构建上下文（不含 Dockerfile）         │
│                                                                         │
│     忽略它可以：                                                         │
│     1. 减小上下文体积                                                    │
│     2. 修改 Dockerfile 不会导致 COPY . . 的缓存失效                      │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### （2）验证 .dockerignore 是否生效
```bash
# 方法1：查看构建上下文大小

Sending build context to Docker daemon  52.4MB   # ← 看这个数字
                                        ↑
                              应该比项目目录小很多

# 方法2：列出实际会发送的文件（调试用）
$ docker build -f /dev/null -t test . 2>&1 | head
# 或者用这个小技巧
$ tar -czf - . | wc -c   # 不忽略的大小
```

## 2.Docker辅助开发流程
### （1）原理
- 在 docker-compose.dev.yml 中配置好第三方服务的"服务端"
  - **端口映射**：`宿主机端口:容器端口`，让宿主机程序能访问容器服务
  - **网络配置**：容器间可通过"服务名"互相访问（宿主机程序用不到）

- 通过 docker-compose 命令运行，将服务端在容器中启动

- **宿主机的 C++ 程序连接服务时**：
  - **连接地址**：`127.0.0.1:映射的宿主机端口`
  - 宿主机收到请求后，通过**端口映射转发**到容器内的服务

### （2）客户端下载“客户端库”
**客户端库仍然需要安装**
- Docker 容器里运行的是"服务端程序"（mysqld、redis-server）
- 你的 C++ 程序需要"客户端库"（libmysqlclient、hiredis）来连接这些服务
- 如果在宿主机编译，宿主机需要安装这些客户端库
```bash
# 宿主机仍然要装客户端库（编译用）
sudo apt install libmysqlclient-dev   # MySQL 客户端库
sudo apt install libhiredis-dev       # Redis 客户端库

# Docker 只负责运行服务端
docker compose up -d   # 启动 MySQL、Redis 服务端
```

### （3）docker-compse.dev.yml的编写与使用
```bash

    image: mysql:8.0
    ports:
      - "3306:3306"
    environment:
      MYSQL_ROOT_PASSWORD: 123456
    volumes:
      - mysql_data:/var/lib/mysql
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 3
    networks:                       # 指定该服务加入的网络
      - dev-network

  redis:                            # 示例：添加 Redis 服务
    image: redis:7-alpine
    ports:
      - "6379:6379"
    networks:
      - dev-network

  dev-network:
    driver: bridge                  # 默认桥接模式
```
