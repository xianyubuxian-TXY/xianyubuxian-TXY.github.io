# 一、分布式系统
## 1.分布式系统概述
- **分布式系统 = 多台机器通过网络协作，对外表现为一个统一系统。** 
- 它是一个广泛的技术概念，涵盖**计算、存储、数据库、消息、缓存、协调**等多个领域。
- 分布式带来了性能、容量、可靠性的提升，但也引入了网络不可靠、数据一致性、部分失败等核心挑战，CAP 定理和 BASE 理论是理解这些挑战的理论基础。
```bash
单机系统：
  用户 ──→ [一台服务器搞定一切]

分布式系统：
  用户 ──→ [看起来是一个系统]
              │
              ├── 机器A（处理一部分）
              ├── 机器B（处理一部分）
              ├── 机器C（处理一部分）
              └── 机器D（处理一部分）
              
  用户感知不到背后有多少台机器
```

## 拓展：RPC在分布式系统中的角色
```bash
RPC 在分布式系统中的角色：

  ✅ 微服务/应用层 → RPC 是主角（gRPC / Dubbo）
  ✅ 部分中间件内部 → 也用 RPC（TiDB 内部用 gRPC）
  ❌ 异步场景 → 消息队列更合适
  ❌ 很多中间件内部 → 自定义协议更高效
  ❌ 集群管理 → Gossip / Raft 等专用协议
  ❌ 对外 API → HTTP/REST 更通用

总结：RPC 是分布式系统通信的重要形式之一，
     在应用层（微服务）是最主流的选择，
     但在整个分布式技术栈中，只是众多通信方式之一。
```



# 二、微服务
- 微服务是分布式的一种应用形式，但分布式远不止微服务

## 1.微服务介绍
- **微服务的本质**：
	- 逻辑上独立的服务，部署在**不同的进程/机器上**，通过 gRPC（或其他协议）进行网络通信。
	- **微服务 = 拆成独立进程 + 通过网络通信协作，RPC 是通信的核心技术。**
	- **RPC参考**：{% post_link "C++/C++后端开发常用库/protobuf的使用" "protobuf的使用" %} 中 "RPC（远程过程调用）"
- **微服务的底层原理**：
	- 把原来同一进程内的函数调用，变成了跨网络的远程调用。
	- RPC（尤其是 gRPC）是实现这种远程调用的主流方式——它通过 Stub 自动生成 + Protobuf 序列化 + HTTP/2 传输，让远程调用看起来和本地调用一样简单。
	- 但微服务通信不只有 RPC，还有消息队列（异步）、REST（对外 API）等，实际项目中是混合使用的。



## 2.实现微服务项目的两种方式

- 后面都将以：gRPC分布式框架实现的微服务用户管理系统为例（包含auth、user独立服务）

### （1）单仓库，部署时选择性注册
- 在同一个项目中通过gRPC框架实现所有服务，然后在不同的服务器上运行该项目，但只是注册的服务不同
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260316163059899.png)![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260316163143235.png)

### （2）独立仓库，各服务完全独立（更标准的微服务做法）
- **独立 ≠ 孤立**,**微服务模块之间要通信**
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260316163224931.png)

### 补充：独立 ≠ 孤立
- **独立**指的是：独立开发、独立部署、独立扩展、独立内存空间。
- 不是指它们互不往来——恰恰相反，微服务之间的**通信**是整个架构的核心。
- **示例：用户下单购买商品**
```bash
客户端: "我要下单"
    │
    ▼
┌──────────┐   ① 验证 token，这人是谁？    ┌──────────┐
│  Order   │ ──────────────────────────→  │  Auth    │
│  Service │ ←────────────────────────── │  Service │
│          │   "user_id=42, 有效"         │          │
│          │                              └──────────┘
│          │
│          │   ② 查一下这个用户的收货地址     ┌──────────┐
│          │ ──────────────────────────→  │  User    │
│          │ ←────────────────────────── │  Service │
│          │   "北京市朝阳区..."           │          │
│          │                              └──────────┘
│          │
│          │   ③ 扣库存                    ┌──────────┐
│          │ ──────────────────────────→  │ Inventory│
│          │ ←────────────────────────── │  Service │
│          │   "库存足够，已扣减"           │          │
│          │                              └──────────┘
│          │
│          │   ④ 扣钱                      ┌──────────┐
│          │ ──────────────────────────→  │ Payment  │
│          │ ←────────────────────────── │  Service │

一个"下单"动作，涉及 5 个服务之间的 5 次通信。
没有任何一个服务能独自完成这件事。
```

## 3.微服务独立仓库下的公共代码复用问题
- 在”独立仓库“中，各服务完全独立，但每个服务都需要：数据库连接、日志、配置加载、gRPC 拦截器、错误处理、Proto 生成代码……如果每个仓库都复制一份，维护噩梦。
- **解决方案：把公共代码抽成独立的共享库。**

### （1）哪些代码会重复
```bash
auth-service/                    user-service/
├── main.py                      ├── main.py
├── service.py                   ├── service.py
│                                │
│  以下全是重复的 ↓               │  以下全是重复的 ↓
├── db.py          ← 重复！      ├── db.py
├── logger.py      ← 重复！      ├── logger.py
├── config.py      ← 重复！      ├── config.py
├── middleware/     ← 重复！      ├── middleware/
│   ├── auth_interceptor.py      │   ├── auth_interceptor.py
│   └── logging_interceptor.py   │   └── logging_interceptor.py
├── proto/         ← 重复！      ├── proto/
│   ├── auth_pb2.py              │   ├── auth_pb2.py
│   └── user_pb2.py              │   └── user_pb2.py
└── utils/         ← 重复！      └── utils/
    ├── retry.py                     ├── retry.py
    └── tracing.py                   └── tracing.py
```

### （2）四种主流解决方案
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260316172952918.png)
#### <1>共享库（最常用）
- **把公共代码抽成一个独立的包，各服务通过包管理器引入。**
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260316172151087.png)

#### <2>Monorepo（单仓库多服务）
- 所有服务放在同一个仓库，但各自独立构建、独立部署。
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260316172759159.png)

#### <3>代码生成 / 脚手架模板
- 用模板工具自动生成每个新服务的骨架代码。
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260316172857575.png)

#### <4>Sidecar / 基础设施下沉（Service Mesh）
- 把公共功能从代码中彻底移出，变成独立的基础设施。
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260316172935887.png)
