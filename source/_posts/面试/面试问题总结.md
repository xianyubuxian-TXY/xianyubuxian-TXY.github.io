### （1）架构图

![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260318233848677.png)

### （2）项目难点、亮点、瓶颈、优化
#### <1>项目难点
- **如何解决跨服务器聊天问题**：登录后按 userid 订阅 Redis channel，单聊/群聊都要判断“本机在线、其他节点在线、离线”三种路径

#### <2>项目亮点
- **单例模式**
- **Nginx反向代理，实现负载均衡**
- **redis的发布订阅模式，解决跨服务器聊天问题**
- **数据库连接池，避免频繁创建、释放连接，实现连接的复用**

#### <3>项目瓶颈
- **最大瓶颈是群聊大锁**：groupChat 持有 \_connMutex 后还会做 MySQL 查询、Redis 发布、离线消息落库，临界区太大，群越大越容易串行化。
- **协议层没真正解决 TCP 粘包拆包**：当前是 retrieveAllAsString + json::parse，这是源码里的潜在风险点。

#### <4>项目优化
- **已做优化**：Muduo 多线程 Reactor、Redis 独立订阅线程、MySQL 连接池配置、Nginx TCP 负载均衡、Linux 和 MySQL 参数调优。
- **协议改成长度前缀 + Protobuf**

### （3）性能指标是如何测试得到的
- 通过写压测脚本，在本地 Linux 环境下做并发压测得到，测试重点主要分为并发长连接、登录 QPS、延迟指标几个部分。
- 这些线程**并不是通过 barrier 严格做到“同一时刻同时起跑”**，而是主线程快速创建出这 100 个 worker 线程，让它们在同一个压测窗口里并发发请求


#### <1>并发长连接 5W+
- 通过创建大量**并发客户端**，使用多线程同时向服务器发起 **TCP 连接**和**登录请求**，统计最终成功建立并保持的连接数量。连接建立后不会立即关闭，而是保活一段时间，并通过心跳机制检查这些连接在一段时间内是否仍然稳定，从而评估最大并发连接数和连接稳定性。为了支持 5W+ 长连接，我还结合 Linux 网络参数和服务器参数做了调优，比如文件描述符上限、Mysql连接池大小、端口范围等。
    - 压测端用的是“**固定数量线程 + 每个线程批量建连**”的方式：max_connection_test 默认并发线程数是 50，然后把目标连接数平均分给这些线程，比如测 50000 个连接时，如果还是 50 个线程，那每个线程大约负责创建 1000 个连接，而**不是一个TCP连接一个线程**。
    - 每个线程是在循环里依次做 connect + login，成功后把 fd 保存起来，然后测试程序统一持有这些连接，等待 10s 后再通过 MSG_PEEK 做存活探测，统计还有多少连接是活的，所以它更准确地说是“**少量线程批量创建大量长连接，再观察稳定性**”。

##### 1）负载均衡方面
- 使用**Nginx实现负载均衡**：因为Nginx反向代理时，需要同时与客户端、目标服务器建立TCP连接，为了达到5w+并发连接，需要进行一定的Nginx配置
    - 首先：就是需要将“worker进程文件描述符限制”调大，避免建立TCP连接时没有文件描述符可用
        - 参考：{% post_link "C++/C++后端开发常用库/nginx的使用" "nginx的使用" %}
    - 其次：“事件模型”选择 epoll模型、设置“允许一次接受多个连接（worker_connections）”、“最少连接负载均衡策略(least_conn)”、“长连接超时（proxy_timeout）”等

##### 2）性能调优方面
- 主要包括“系统性能调优”、池化技术调优
- **系统性能调优**：
    - 通过**ulimit设置文件描述符限制**到200000，避免多并发连接时文件描述符不足
    - 进行**TCP参数优化**：
```bash
sudo sysctl -w net.core.somaxconn=65535                     # 监听队列最大长度 （全连接队列）
sudo sysctl -w net.core.netdev_max_backlog=65535            # 网卡接收队列最大长度
sudo sysctl -w net.ipv4.tcp_max_syn_backlog=65535           # 半连接队列最大长度（SYN_RECV状态）
sudo sysctl -w net.ipv4.tcp_fin_timeout=10                  # FIN_WAIT_2 状态超时时间（秒）
sudo sysctl -w net.ipv4.tcp_tw_reuse=1                      # 允许复用 TIME_WAIT 状态的连接
sudo sysctl -w net.ipv4.ip_local_port_range="1024 65535"    # 本地端口分配范围
```
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260308221210660.png)
- **池化技术调优**：
    - 将muduo线程池大小扩大一倍
    - 在配置文件中，将mysql连接池大小增加到150，增加最小空闲连接等
    - 增加mysql最大连接数，适配mysql连接池大小

##### <2>登录 QPS
- 通过压测程序提前建立大量连接，然后多个并发客户端持续向服务器发起登录请求，统计单位时间内登录成功的请求数，比如统计 1 秒 内成功处理了多少次登录请求，这个值就是登录 QPS。
    - 登录 QPS 不是单线程测的，也不是要求所有请求绝对同时起跑，而是通过**设置固定并发度**，比如 100 个工作线程，在同一时间窗口内并发向服务器发起登录请求，持续统计单位时间内成功处理的登录数，从而得到 QPS。这个项目里既有批次并发的方式，也有持续并发施压的方式。
    - connection_test 的做法是按批次启动 concurrency 个线程，每个线程做一次 connect + login，这一批线程一起跑，结束后再跑下一批，所以是“批内近似同时发起”。默认并发数是 100

##### <3>延迟指标 P99
- **对每一次请求都记录开始时间和结束时间**，得到**单次请求耗时**。等压测结束后，把所有**请求时延从小到大排序**，取位于后 99% 的时延作为 P99。比如发起 1000 个请求，排完序后看第 990 个请求对应的耗时，就可以近似理解为 P99。
    - 总请求数由 totalRequests 决定，默认 10000
    - 每个线程负责的请求数大约是 totalRequests / concurrency
    - **比如 100 线程、10000 请求，就是每个线程大约发 100 次登录请求。**

## 3.user_service

### （1）架构图
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260319100116816.png)

### （2）主要瓶颈
- **连接池规模偏小**
- **gRPC 现在是同步阻塞模型**
- **服务拆分还不彻底**
- **目前只支持统一服务器不同端口的分布，如果要分布的不同服务器，需要解决数据同步问题**

### （3）可优化处
#### <1>数据三副本
- **MySQL**：
    - 所有服务实例连接**同一个逻辑数据库集群**，不是每台机器各自一份数据库
    - 写请求只进主库，读请求可以按场景走从库（**主从复制**）
    - **同步依赖数据库复制机制**，比如 binlog + GTID，不是应用自己同步
- **Redis**：
    - 方案一：**1 主 2 从 + 3 Sentinel**
    - 方案二：如果后面数据量上来，用 **Redis Cluster**
    - 验证码、登录失败计数、session/token 这类共享状态都放共享 Redis，不放本地内存

#### <2>服务节点集群怎么做
- 推荐至少 3 个 auth/user 服务实例，分布在不同主机。
    - 每个实例都是无状态服务
    - 启动后注册到 ZooKeeper
    - 注册信息里带 host/port/weight/version/zone
    - 客户端从本地实例缓存里选节点
    - 支持权重路由、同机房优先、灰度发布

#### <3>熔断机制怎么设计
- **状态机**：
    - Closed 正常放行
    - Open 直接拒绝该实例请求
    - Half-Open 放少量探测流量

### （2）mmap + write
- 文件内容直接映射到进程地址空间：使得进程可以像访问内存一样读写文件，而无需通过 read/write 等系统调用。
```cpp
#include <sys/mman.h>

void *mmap(
    void   *addr,   // 建议映射到哪个虚拟地址（通常传 NULL，让内核自己选）
    size_t  length, // 映射的字节长度
    int     prot,   // 内存保护标志：
                    //   PROT_READ  → 可读
                    //   PROT_WRITE → 可写
                    //   PROT_NONE  → 不可访问
    int     flags,  // 映射类型：
                    //   MAP_SHARED  → 共享映射（修改会写回文件）
                    //   MAP_PRIVATE → 私有映射（修改不影响文件，写时复制）
    int     fd,     // 要映射的文件 fd
    off_t   offset  // 从文件的哪个偏移量开始映射（必须是页大小的整数倍）
);

// 返回值：成功返回映射区域的起始虚拟地址，失败返回 MAP_FAILED（即 (void*)-1）
```
```cpp
int main() {
    // 1. 打开文件
    int fd = open("test.txt", O_RDWR | O_CREAT, 0644);

    // 2. 获取文件大小
    struct stat stat_buf;
    fstat(fd, &stat_buf);
    size_t file_size = stat_buf.st_size;

    // 3. 建立内存映射
    char *map_ptr = mmap(NULL, file_size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
    if (map_ptr == MAP_FAILED) { // 错误处理 }

    // 4. 像操作内存一样修改文件（自动同步到底层文件）
    printf("Original content: %s\n", map_ptr);
    map_ptr[0] = 'h'; // 修改第一个字符为小写

    // 5. 可选：主动同步修改到磁盘（内核也会自动同步）
    msync(map_ptr, file_size, MS_SYNC);

    // 6. 解除映射并关闭文件
    munmap(map_ptr, file_size);
    close(fd);
    return 0;
}
```

#### <1>二进制指令执行过程（基础知识）
- 指令执行过程：**取指、译码、执行、访存、写回**（流水线加速）

##### 1）取指
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260310153733766.png)

##### 2）译码
- 把取指阶段拿到的「二进制指令码」翻译成CPU能理解的"操作命令"：
    - 指令的二进制编码包含两部分核心信息：**操作码（做什么） + 操作数（对谁做）**；
    - 译码阶段会解析出：要执行的操作（比如加法、减法、移动数据）、操作数的来源（比如寄存器、内存地址、立即数）。

##### 3）执行
- 执行译码阶段解析出的**具体操作**，是指令的"计算核心阶段"：
    - 如果是算术运算（加减乘除）、逻辑运算（与或非）：由运算器完成计算；
    - 如果是数据移动（比如把A寄存器的值移到B寄存器）：直接在寄存器间传输；
    - 如果是控制操作（比如跳转）：修改程序计数器（PC）的值，改变指令执行顺序。

##### 4）访存
- 与内存交互：读取**内存中**的数据，或**把执行结果写入内存**（**不是所有指令都需要这一步**）：
    - 读内存：比如指令要求"把内存地址0x100的值加载到寄存器"，执行阶段算出地址后，访存阶段从内存取数；
    - 写内存：比如指令要求"把ALU的计算结果存入内存地址0x200"，访存阶段完成写入；
    - 无内存操作的指令（比如寄存器间的加法）：这一步会直接跳过，仅做"空操作"。
                                                             
##### 5）写回
- 把指令执行的最终结果**写回CPU的寄存器**（或特殊存储位置），供后续指令使用：
    - 如果是算术运算：把ALU的结果写入目标寄存器；
    - 如果是读内存：把从内存取出的数据写入目标寄存器；
    - 写回是指令执行的"收尾"，确保结果被保存，后续指令能调用。

    - **2.创建虚拟地址空间**：创建“VMA”，记录“虚拟地址空间”到“ELF文件空间”的映射关系；创建“页表”，记录“物理空间（内存空间）”到“虚拟地址空间”的映射关系

#### <2>为什么是三次握手，而不是两次？
- 1.**避免无效连接浪费资源**:
    - **若为两次握手，延迟的旧SYN包会让服务端建立无效连接并分配资源，且无法快速回收，浪费资源** ——>（如A向B发送SYN包后，由于网络阻塞，一直没到达B，然后A超时重传再次传送一个SYN包，A与B建立连接，当A与B断开连接后，之前阻塞的SYN包到达B，如果只是两次连接，B收到后就会立即进入“连接状态“，而这个连接是无效的。）
- 2.**完成双向序列号同步**
    - **TCP 是全双工通信，需要双方确认彼此的初始序列号**。 ——> 三次握手的三次交互能确保 A 和 B 都知道 “对方收到了自己的序列号”，为可靠传输提供保障。

### （2）四次挥手
#### <1>流程
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260321174054795.png)
```bash
1. 主动关闭方（通常是客户端）发送FIN报文（表示“我无数据要发送了”）→ 主动关闭方进入 FIN_WAIT_1 状态；

2. 被动关闭方（通常是服务端）接收FIN后，立即回复ACK报文 → 主动关闭方收到ACK后进入 FIN_WAIT_2 状态；
   被动关闭方进入 CLOSE_WAIT 状态（半关闭：主动方→被动方的数据流已关闭，被动方仍可向主动方发数据）；

3. 被动关闭方处理完剩余数据后，发送自己的FIN报文 → 被动关闭方进入 LAST_ACK 状态；

4. 主动关闭方接收FIN后，回复ACK报文 → 主动关闭方进入 TIME_WAIT 状态；被动关闭方收到ACK后，
   立即进入 CLOSED 状态；主动关闭方等待 2MSL 后，进入 CLOSED 状态。
```

#### <2>为什么 FIN 需要四次不能三次？
- **因为TCP连接是全双工通信，需要两端都关闭连接** ——> 两次报文关闭一个方向的数据流，四次报文才能关闭全双工的两个方向


#### <3>TIME_WAIT状态的作用
- **防止最后一个 ACK 丢失**：被动关闭方（如服务端）在发送 FIN 包后进入 LAST_ACK 状态，等待主动关闭方（如客户端）回复最后一个 ACK。若该 ACK 因网络波动丢失，被动关闭方会超时重传 FIN 包。主动关闭方发送 ACK 后进入 TIME_WAIT 状态，持续 2MSL 时长，刚好能接住被动关闭方重传的 FIN 并重新回复 ACK，确保被动关闭方能正常关闭连接。同时，2MSL 也能让网络中旧连接的滞留报文完全失效，避免干扰后续四元组相同的新连接。

#### <1>net.core.somaxconn = 65535
```bash
作用：全连接队列（Accept Queue）的最大长度上限

全连接队列是什么？
  三次握手完成后，连接进入 ESTABLISHED 状态
  → 放入全连接队列，等待应用程序调用 accept() 取走

实际队列长度 = min(somaxconn, listen() 的 backlog 参数)

  // 应用代码
  listen(sockfd, 128);    ← backlog = 128
  
  实际长度 = min(65535, 128) = 128
  → 所以调大 somaxconn 后，应用代码的 backlog 也要调大！
  → Nginx: listen 80 backlog=65535;

队列满了会怎样？
  新完成三次握手的连接被直接丢弃（RST 或静默丢弃）
  客户端会看到 "Connection reset" 或超时

什么时候会满？
  accept() 太慢（应用处理不过来）
  突发大量并发连接（秒杀、突发流量）

监控命令：
  ss -lnt    ← 看 Send-Q（队列上限）和 Recv-Q（当前积压）
  
  State    Recv-Q  Send-Q  Local Address:Port
  LISTEN   0       128     *:80                ← Send-Q=128 是上限
  LISTEN   129     128     *:80                ← Recv-Q > Send-Q → 溢出！
```

```bash
作用：网卡接收队列（per-CPU backlog queue）的最大长度

注意：这个和"全连接队列"不是一回事！

            ┌───────────────┐

            └───────┬───────┘
                    ↓
              协议栈处理（TCP/IP）

什么时候会满？

  高流量场景（10Gbps+），CPU 来不及处理

满了会怎样？
  直接丢包！在协议栈之前就丢了
  → 应用层无感知，只能通过内核计数器发现

监控：

  第二列 = 因 backlog 满而丢弃的包数

```

#### <3>net.ipv4.tcp_max_syn_backlog=65535
```bash
作用：半连接队列（SYN Queue）的最大长度

半连接队列是什么？
  收到 SYN，回了 SYN+ACK，等待对方 ACK
  此时连接处于 SYN_RECV 状态 → 放在半连接队列中

  客户端 ── SYN ──→ 服务端
                     ↓
              放入半连接队列（SYN_RECV）
                     ↓
  客户端 ←─ SYN+ACK ─ 服务端
                     
  等待 ACK ...

队列满了会怎样？
  新来的 SYN 被丢弃 → 客户端看到连接超时
  → 这就是 SYN Flood 攻击的原理！

  SYN Flood 攻击：
    攻击者发大量伪造源 IP 的 SYN 包
    → 半连接队列被填满
    → 正常用户无法建立连接

防御措施（配合使用）：
  net.ipv4.tcp_syncookies = 1   ← 开启 SYN Cookie
  
  SYN Cookie 原理：
    半连接队列满后，不再存 SYN_RECV 状态
    而是把连接信息编码到 SYN+ACK 的序列号中
    收到 ACK 时解码验证 → 直接进入 ESTABLISHED
    → 不占半连接队列空间！

监控：
  netstat -s | grep "SYNs to LISTEN"
  cat /proc/net/tcp   （状态 SYN_RECV = 0x03）

```

#### <4>net.ipv4.tcp_fin_timeout=10
- TCP连接完全关闭，需要**四次挥手**，每两次握手，表示一方关闭了数据流 ——> 如果一方主动关闭，两次挥手后进入FIN_WAIT_2,但另一方迟迟不发FIN进行后两次挥手，则连接卡在 FIN_WAIT_2 状态 ——> 设置tcp_fin_timeout参数，FIN_WAIT_2 状态的超时时间，超时后内核强制回收这个连接
```bash
作用：FIN_WAIT_2 状态的超时时间

默认值：60 秒
设为：10 秒

FIN_WAIT_2 是什么？

  主动关闭方（通常是服务端）：

  服务端             客户端
    │── FIN ──→│
    │          │  服务端进入 FIN_WAIT_1
    │←── ACK ──│
    │          │  服务端进入 FIN_WAIT_2  ← 等对方发 FIN
    │          │
    │  如果客户端迟迟不发 FIN（程序bug/崩溃）...
    │  连接卡在 FIN_WAIT_2 状态
    │  
    │  tcp_fin_timeout 秒后 → 内核强制回收这个连接

为什么要调小？
  FIN_WAIT_2 的连接占用内核资源（文件描述符、内存）
  高并发服务器上可能积累大量 FIN_WAIT_2 连接
  60 秒太长 → 10 秒足够正常客户端完成关闭

风险：
  如果网络延迟大（跨国连接），10 秒可能太短
  → 连接被提前回收 → 对方发的 FIN 到了却找不到连接 → RST
  → 一般内网或低延迟环境设 10~15 秒没问题

```


#### <5>net.ipv4.tcp_tw_reuse=1
```bash
作用：允许复用 TIME_WAIT 状态的连接（仅对 客户端/主动连接方 生效）

TIME_WAIT 是什么？
  主动关闭连接的一方，发完最后一个 ACK 后进入 TIME_WAIT
  默认等待 2MSL（通常 60 秒）后才释放

  为什么要等 2MSL？
    ① 确保最后一个 ACK 能到达对方
    ② 确保网络中残留的旧数据包消亡，不会干扰新连接

问题：高并发短连接场景
  例：HTTP 短连接，服务器主动关闭
    → 每秒处理 10000 个请求
    → 每个 TIME_WAIT 持续 60 秒
    → 最多积累 600,000 个 TIME_WAIT 连接！
    → 端口耗尽，无法建立新连接

tcp_tw_reuse = 1 的效果：
  允许新的出站连接（connect()）复用 TIME_WAIT 状态的端口
  前提：TCP 时间戳选项开启（tcp_timestamps = 1，默认开启）
  
  复用条件：
    新连接的时间戳 > 旧连接的时间戳 → 可以安全区分新旧数据包

注意事项：
  ✅ 只影响主动连接方（客户端角色、反向代理连后端）
  ❌ 不影响 listen 端的 accept
  ✅ 比已废弃的 tcp_tw_recycle 安全得多（NAT 环境不会出问题）
  
  tcp_tw_recycle（已废弃，Linux 4.12+ 移除）：
    在 NAT 环境下会导致大量连接失败 → 千万别用
```

### （3）流量控制
- **核心目的：** 匹配发送方与接收方处理能力，避免接收缓冲区溢出，保障数据可靠传输。
- **核心机制：** 接收端维持一个”接收窗口“（rwnd=RecvBuffer-[LastByteRcvd-LastByteRead]），在回复ACK时会将该窗口大小一起发给发送方，让发送方发送的字节数不能大于接受窗口大小，从而实现流量控制
  - RecvBuffer：接收缓存；LastByteRcvd：已写入接收缓存的数据流的最后一个字节编号；LastByteRead：已经被应用层读取的最后一个字节编号
- **特殊情况：** 当 rwnd=0 时，发送方会停止发送新数据，若接收方后续消费了缓冲区数据、更新了 rwnd，却无法通过常规 ACK 告知发送方（因为没有数据传输触发 ACK），会导致发送方一直暂停。
  **TCP 协议的解决方案是：** 发送方在 rwnd=0 时，会启动一个持续计时器（persist timer），当计时器超时后，发送方会主动发送一个一个字节数据的报文段。接收方收到该报文后，必须回复 ACK，并携带最新的 rwnd 值。

### （4）拥塞控制
- **核心目的：** 让发送方的传输速率匹配当前网络的承载能力，避免因发送速率过高导致网络中出现大量报文积压、丢包、延迟激增等拥塞问题
- **核心机制：** 发送方维持一个拥塞窗口cwnd，对发送发送流量的数据进行限制，同时结合丢包 / 时延反馈判断网络状态
- **四个阶段**：
  - **1.慢启动：** 连接建立初期，cwnd的值为的值设置为1MSS（最大报文段长度），之后每个RTT，cwnd就翻倍（指数增长）。何时结束？
    - **1）** 出现超时指示的丢包事件（即拥塞）：发送方将cwnd设置为1重新开启慢启动过程，并将ssthresh（”慢启动阈(yu)值“）设置为cwnd/2。（第一次慢启动可以时ssthresh可以设置为无穷大）
    - **2）** 直接与ssthresh相关：当cwnd的值等于ssthresh时，结束慢启动进入拥塞避免模式
    - **3）** 如果检测到3个冗余ACK：TCP执行快速重传并进入快速恢复状态
  - **2.拥塞避免**：当cwnd的值等于ssthresh时，进入拥塞避免，之后每个RTT,cwnd+1MSS。何时结束？
    - **1）** 出现丢包：与慢启动同
    - **2）** 检测到3个冗余ACK：同慢启动
  - **3.快速重传：** 不通过”超时“指示丢包，而是”由3个冗余的ACK事件“指示丢包，避免cwnd大幅缩减，进入快速恢复状态
  - **4.快速恢复：**
    - **1）** 将 ssthresh 设为当前 cwnd 的一半；
    - **2）** 将 cwnd 设为 ssthresh + 3（补偿已收到的 3 个重复 ACK）；
    - **3）** 之后每收到一个冗余ACK，cwnd+1MSS（注意与拥塞避免的区别：拥塞避免是每个RTT，cwnd+1MSS）
    - **4）** 当收到1个新的ACK（非冗余），将cwnd设为ssthresh，进入拥塞避免

### （5）TCP 粘包拆包的本质与解决方案
- **1.本质**：TCP是流式协议特性，无天然消息边界
- **2.解决方案：**
  - **1）分隔符方案（简单易实现）**
    - 如像http协议，在每个消息的末尾添加特殊分隔符'\\r\\n\'
  - **2）TLV格式（Type-Length-Value，最常用）**
    - 每个消息分为三部分（Type,Length，Value）组成，如RPC二进制协议
  - **3） 固定长度方案（最简单但灵活性差）**

    - 约定所有消息的长度固定（如每个消息都是 100 字节）
  - **4）Protobuf Varint 编码（高效灵活）**

    - 利用 Protobuf 的 Varint（可变长度整数）编码，消息本身自带长度信息，无需额外定义`Length`字段。Protobuf 会自动序列化消息，并在接收端反序列化时识别完整消息边界。

## 2.介绍一下epoll多路复用

### （0）一句面试版
```text
epoll 是 Linux 下高性能 I/O 多路复用机制。它通过 epoll_create(1)/epoll_create1 创建实例，用 epoll_ctl 管理关注的 fd 及事件，用 epoll_wait 获取就绪事件。底层是“红黑树管理关注集合 + 就绪链表管理活跃 fd + 回调通知”，避免了 select/poll 每次轮询全量遍历。触发模式有 LT（默认，易用）和 ET（高效但必须非阻塞并读到 EAGAIN）
```

### （1）什么是epoll？
- epoll 是 Linux 内核提供的一种**高效的 I/O 多路复用机制**，用于监控多个文件描述符（fd）上的 I/O 事件。
- 它是 select 和 poll 的增强版本，能够在大量并发连接中高效工作。

### （2）epoll的核心API（三个系统调用）
```cpp
// 1.epoll把用户关心的文件描述符上的事件放在一个“内核事件表”中
//   - epoll_create 用来创建唯一标识epoll“内核事件表”的文件描述符
int epoll_create(int size);  // size 参数已被忽略，但必须 > 0

// 2.操作epoll的内核事件表，将用户关心的文件描述符从内核事件表中“注册/修改/删除”
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
//   op: EPOLL_CTL_ADD / EPOLL_CTL_MOD / EPOLL_CTL_DEL

// 3. 在超时时间内等待内核事件表上的事件发生，函数成功时返回就绪的文件描述符个数
//  - 出参 epoll_event数组中存储了发生事件的epoll_event,从中可提取fd
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);
```
- epoll_wait 不一定阻塞
    - timeout = -1：一直阻塞
    - timeout = 0：立即返回（非阻塞轮询）
    - timeout > 0：最多阻塞 timeout 毫秒

### （3）epoll的工作流程
- 通过epoll_create创建唯一标识epoll内核事件表的文件描述符
- 创建要监听的文件描述符的epoll_event进行绑定，并设置监听的事件（读事件/写事件/异常事件等）
- 通过epoll_ctl向epoll内核事件表中注册要监听的文件描述符对应的epoll_event
- 调用epoll_wait等待内核事件表上事件的发生，根据返回的“就绪文件描述符”数量，从出参epoll_event数组中取出文件描述符一次进行处理

### （4）epoll的两种工作模式
#### <1>LT（Level Triggered）— 水平触发（默认）
```cpp
// 只要缓冲区有数据可读，每次 epoll_wait 都会通知
event.events = EPOLLIN;  // 默认 LT 模式
```
- 缓冲区**有数据就一直通知**
- 可以不一次性读完，下次还会通知
- **编程简单，不容易丢数据**
- **性能较低**

#### <2>ET（Edge Triggered）— 边缘触发
```cpp
// 只有状态变化时才通知（从无数据 → 有数据）
event.events = EPOLLIN | EPOLLET;
```
- 只在**状态变化时通知一次**
- **必须一次性读完所有数据（必须循环读/写（以及 accept）直到返回 EAGAIN，否则可能丢后续通知。）**（循环读到 EAGAIN）
- **必须配合非阻塞 I/O** 使用
- **性能更高**，减少了 epoll_wait 的调用次数
```cpp
// ET 模式下的典型读取方式
while (1) {
    int n = read(fd, buf, sizeof(buf));
    if (n == -1) {
        if (errno == EAGAIN) break;  // 数据读完了
        perror("read error");
        break;
    }
    if (n == 0) { /* 连接关闭 */ break; }
    // 处理数据...
}
```

### （1）Reactor模型是什么
- **Reactor 是一种事件驱动的网络编程模型。**
    - **事件驱动**：程序不是按固定顺序一直往下执行，而是先“等事件”，事件发生后再执行对应处理逻辑。

    - 哪个**连接就绪**，就把事件分发给对应的 工作线程 去处理

#### <2>多继承需要解决的两个关键问题
##### 1）名冲突
- 在多继承中，当多个基类中包含同名的成员时，在派生类中就会出现**名冲突**
- 解决方法：**基类名受限访问**
```cpp
#include <iostream>
using namespace std;

class A {
public:
    void func() { cout << "A::func()" << endl; }
    int value = 10;
};

class B {
public:
    void func() { cout << "B::func()" << endl; }
    int value = 20;
};

class C : public A, public B {
    // 继承了 A::func()、A::value
    // 继承了 B::func()、B::value
    // 名冲突！
};

int main() {
    C c;
    
    // ❌ 编译错误：二义性
    // c.func();      // error: ambiguous
    // c.value = 5;   // error: ambiguous
    
    return 0;
}
```
```cpp
#include <iostream>
using namespace std;

class A {
public:
    void func() { cout << "A::func()" << endl; }
    int value = 10;
};

class B {
public:
    void func() { cout << "B::func()" << endl; }
    int value = 20;
};

class C : public A, public B {
public:
    // 方法1：在派生类中用作用域解析符调用
    void callBoth() {
        A::func();  // 明确调用 A 的版本
        B::func();  // 明确调用 B 的版本
    }
    
    // 方法2：在派生类中定义自己的版本（隐藏基类版本）
    void func() {
        cout << "C::func() - 调用自己的实现" << endl;
        // 如需调用基类，仍可用 A::func() 或 B::func()
    }
};

int main() {
    C c;
    
    // ✅ 使用基类名受限访问
    c.A::func();      // 输出: A::func()
    c.B::func();      // 输出: B::func()
    
    c.A::value = 100;
    c.B::value = 200;
    cout << "A::value = " << c.A::value << endl;  // 100
    cout << "B::value = " << c.B::value << endl;  // 200
    
    // ✅ 调用派生类自己的方法
    c.func();         // 输出: C::func() - 调用自己的实现
    c.callBoth();     // 输出: A::func()  B::func()
    
    return 0;
}
```

##### 2）重复继承（菱形继承） —— 虚基类
- 在多继承中，如果继承的类有公共的基类，则会出现**重复继承** ——> 公共基类中的数据成员在多继承的派生类中有多个拷贝
- 解决方案：**虚基类**
```cpp
class A{
	int x;
	......
};

class B: public A {......};
class C: public A {......};
class D: public B,public C{
	......
	void f(){
		...B::x... // 访问从B继承来的x
		...C::x... // 访问从C继承来的x
	}
};
......
D d;	// d有两个x
```
- 通常情况下，我们需要把这两个x合并为一个x，此时，应把类A定义为类B和类C的**虚基类**，这样定义后，类D就只有一个成员x了
```cpp
class B: virtual public A {......};
class C: virtual public A {......};
class D: public B,public C{
	......
	void f(){
		...x...	// 访问从B和C继承来的公共x
	}
};
......
D d;	// d只有一个x
```


##### 拓展：包含虚基类的类的“两个注意点”
- 虚基类的构造函数由该类的构造函数直接调用
- 虚基类的构造函数优先于非虚基类的构造函数执行

### （3）多态

## 3.虚函数相关面试问题
### （1）介绍一下虚函数、虚函数表、虚函数指针，以及三者如何协作实现"虚函数动态绑定"
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260305115526861.png)
- **虚函数**： virtual定义的成员函数
- **虚函数表**：对于每一个类，如果它有虚函数（**包括从基类继承来的**），则编译程序将会为其创建一个虚函数表（vtbl），表中记录了该类中所有虚函数的入口地址
	- 虚函数表存储在**只读数据段 .rodata**中
- **虚函数表指针**：当创建一个包含虚函数的类的对象时，在所创建对象的内存空间中有一个隐藏的指针（vptr）指向该对象所属类的虚函数表。
	- **vptr初始化时机**： 在构造函数执行时设置（先指向父类 vtable，再指向子类 vtable）
- **虚函数动态绑定的实现方式**：
	- 基类通过virtual定义成员函数，子类继承该基类后，由于包含虚函数（包括从基类继承来的），故程序会在 **.rodata** 段为该类创建一个**虚函数表**，为该类的所有对象共用。
	- 子类的虚函数表中，最开始存储的是**父类的虚函数的入口地址**，当子类override该虚函数后，虚函数表中对应位置将改为存储**子类override后的虚函数的入口地址**
	- 当通过**引用/指针**访问类的虚函数时，将会利用它们引用或指向的对象内存空间中的**虚函数表指针**从**虚函数表**中获取相应虚函数的入口地址来调用相应的函数

### （2）vptr初始化时机
- 在“构造函数”的**初始化列表之后、函数体之前**被隐式设置。
- **先指向父类 vtable，再指向子类 vtable**

#### <1>为什么在“初始化列表之后、函数体之前”
- 在构造函数中可能调用虚函数
```cpp
class Base {
public:
    Base() { 
        cout << "Base 构造" << endl;
        this->show();  // 调用哪个版本？
    }
    virtual void show() { cout << "Base::show" << endl; }
};

class Derived : public Base {
public:
    Derived() { 
        cout << "Derived 构造" << endl;
        this->show();  // 调用哪个版本？
    }
    void show() override { cout << "Derived::show" << endl; }
};

Derived d;
```

#### <2>为什么“先指向父类 vtable，再指向子类 vtable”
- 因为构造 Derived 对象（子类对象）时，先执行 Base 构造函数（此时 vptr 指向 Base::vtable），再执行 Derived 构造函数（vptr 更新为 Derived::vtable）。
```cpp
创建 Derived 对象 d：

时间线 ──────────────────────────────────────────────────────────►

     ┌─────────────────────────────────────────────────────────┐
     │ 1. 分配内存                                             │
     │    vptr = 未初始化                                      │
     └─────────────────────────────────────────────────────────┘
                              ↓
     ┌─────────────────────────────────────────────────────────┐
     │ 2. 进入 Base::Base()                                    │
     │    vptr = &Base::vtable  ←── 设置为父类的 vtable        │
     │    执行 Base 构造函数体                                  │
     │    调用 this->show() → Base::show() ✓                  │
     └─────────────────────────────────────────────────────────┘
                              ↓
     ┌─────────────────────────────────────────────────────────┐
     │ 3. 进入 Derived::Derived()                              │
     │    vptr = &Derived::vtable  ←── 更新为子类的 vtable     │
     │    执行 Derived 构造函数体                               │
     │    调用 this->show() → Derived::show() ✓               │
     └─────────────────────────────────────────────────────────┘
                              ↓
     ┌─────────────────────────────────────────────────────────┐
     │ 4. 构造完成                                             │
     │    vptr = &Derived::vtable （最终状态）                  │
     └─────────────────────────────────────────────────────────┘
```

### （3）为什么析构函数往往是虚函数，而构造函数不是？
#### <1>析构函数往往是虚函数
##### 1）delete 操作的本质
```cpp
delete p;

// 等价于：
p->~???();           // 第一步：调用析构函数（调用哪个？）
operator delete(p);  // 第二步：释放内存
```
- **非虚析构**：delete p 时，编译器根据指针类型 Base\* 在编译期直接生成调用 Base::~Base() 的代码，不会查虚函数表。
- **虚析构**：析构函数地址存入 vtable，delete p 时通过 vptr → vtable → 析构函数地址，运行时找到对象实际类型的析构函数（Derived::~Derived()）进行调用，然后自动调用父类的析构函数。
- 这就是为什么**有继承关系且可能通过基类指针删除对象时，往往使用虚析构函数**
```cpp
【非虚析构】                           【虚析构】

delete p;                              delete p;
    │                                      │
    ▼                                      ▼
编译期确定                              运行期查表
调用 Base::~Base()                          │
    │                                      ▼
    │                               ┌──────────────────┐
    │                               │ 1. 取 vptr       │
    │                               │ 2. vptr → vtable │
    │                               │ 3. vtable[0]     │
    │                               │    → ~Derived    │
    │                               └──────────────────┘
    │                                      │
    ▼                                      ▼
~Base() 执行                         ~Derived() 执行
    │                                      │
    ▼                                      ▼
operator delete(p)                   然后自动调用 ~Base()
                                           │
                                           ▼
                                    operator delete(p)

结果：Derived::data 泄漏！           结果：正确释放所有资源 ✓
```

## 4.类之间的聚合/组合关系
- **聚合关系**：类似于一个公司，公司员工聚在公司一起工作，公司解散了，员工仍存在，可以去其它公司，不依赖于公司而存在,可以独立存在
- **组合关系**：类似于人与器官，器官依赖于人这个整体，人死了，器官也就死了
```cpp
class A{
	......
};

class B{	// B与A是聚合关系
	A *pm;	// 指向成员对象
public:
	B(A *p) {pm=p;}		// 外部创建，然后传入
	~B() {pm=nullptr;}	// 由外部释放资源，这里只是不再指向传入的对象
};

class C{
	A *pm;
public:
	C() {pm=new A;}		// 由内部创建
	~C(){delete pm;}	// 成员对象随组合类对象消亡
};
......
A *pa=new A;		// 创建一个A类对象
B *pb=new B(pa);	// 创建一个聚合类对象，其成员对象由“外部传入”
C *pc=new C();		// 创建一个组合类对象，其成员对象在组合类对象“内部创建”
......
delete pb;		// 聚合类对象消亡了，其成员对象不再指向pa，且pa并没有消亡，仍可以在其它地方使用
delete pc;		// 组合类对象与其成员对象都消亡
delete pa；		// 聚合类对象原来的成员对象消亡
```

### （3）建造者模式（Builder）⭐⭐⭐⭐
- **核心**：分步骤构建复杂对象，解决"**构造函数参数爆炸**"问题。
```cpp
#include <string>
#include <iostream>

class Person {
public:
    std::string name;
    int         age   = 0;
    std::string email;
    std::string phone;

    void show() const {
        std::cout << "name=" << name
                  << " age="  << age
                  << " email=" << email << "\n";
    }

    // 内部建造者类
    class Builder {
    public:
        Builder& setName(const std::string& v)  { p.name  = v; return *this; }
        Builder& setAge(int v)                  { p.age   = v; return *this; }
        Builder& setEmail(const std::string& v) { p.email = v; return *this; }
        Builder& setPhone(const std::string& v) { p.phone = v; return *this; }

        Person build() { return p; }  // 最终构建，返回完整对象

    private:
        Person p;
    };
};

// 使用：链式调用，可读性强
Person p = Person::Builder()
    .setName("张三")
    .setAge(25)
    .setEmail("zhangsan@example.com")
    .build();
p.show();
```

## 2.结构型模式
###（1）代理模式（Proxy）⭐⭐⭐⭐⭐
- **核心**：控制对原对象的访问，可在访问前后增加逻辑（如权限检查、日志、延迟加载）。
```cpp
#include <iostream>
#include <memory>

// 抽象接口
class IDatabase {
public:
    virtual void query(const std::string& sql) = 0;
    virtual ~IDatabase() = default;
};

// 真实对象
class RealDatabase : public IDatabase {
public:
    void query(const std::string& sql) override {
        std::cout << "[DB] 执行查询: " << sql << "\n";
    }
};

// 代理对象：增加权限检查 + 日志
class DatabaseProxy : public IDatabase {
public:
    DatabaseProxy(std::string user)
        : user_(std::move(user)),
          db_(std::make_unique<RealDatabase>()) {}

    void query(const std::string& sql) override {
        // 前置逻辑：权限检查
        if (user_ != "admin") {
            std::cout << "[Proxy] 权限不足，拒绝访问\n";
            return;
        }
        std::cout << "[Proxy] 日志记录: user=" << user_ << "\n";

        db_->query(sql);  // 转发给真实对象

        // 后置逻辑：计时、审计等...
    }

private:
    std::string                  user_;
    std::unique_ptr<IDatabase>   db_;
};

// 使用
std::unique_ptr<IDatabase> db = std::make_unique<DatabaseProxy>("admin");
db->query("SELECT * FROM users");
// 输出：
// [Proxy] 日志记录: user=admin
// [DB] 执行查询: SELECT * FROM users
```

### （2）装饰器模式（Decorator）⭐⭐⭐⭐
- **核心**：动态地给对象附加功能，比继承更灵活（继承是编译期固定，装饰是运行期组合）。
```cpp
// 抽象组件
class Coffee {
public:
    virtual std::string getDescription() const = 0;
    virtual double      getCost()         const = 0;
    virtual ~Coffee() = default;
};

// 具体组件：基础咖啡
class SimpleCoffee : public Coffee {
public:
    std::string getDescription() const override { return "普通咖啡"; }
    double      getCost()         const override { return 5.0; }
};

// 抽象装饰器（持有一个 Coffee 的引用）
class CoffeeDecorator : public Coffee {
protected:
    std::unique_ptr<Coffee> coffee_;
public:
    explicit CoffeeDecorator(std::unique_ptr<Coffee> c)
        : coffee_(std::move(c)) {}
};

// 具体装饰器：加牛奶
class MilkDecorator : public CoffeeDecorator {
public:
    using CoffeeDecorator::CoffeeDecorator;
    std::string getDescription() const override {
        return coffee_->getDescription() + " + 牛奶";
    }
    double getCost() const override { return coffee_->getCost() + 2.0; }
};

// 具体装饰器：加糖
class SugarDecorator : public CoffeeDecorator {
public:
    using CoffeeDecorator::CoffeeDecorator;
    std::string getDescription() const override {
        return coffee_->getDescription() + " + 糖";
    }
    double getCost() const override { return coffee_->getCost() + 1.0; }
};

// 使用：动态叠加功能
auto coffee = std::make_unique<SimpleCoffee>();
coffee = std::make_unique<MilkDecorator>(std::move(coffee));   // 加牛奶
coffee = std::make_unique<SugarDecorator>(std::move(coffee));  // 加糖

std::cout << coffee->getDescription() << "\n";  // 普通咖啡 + 牛奶 + 糖
std::cout << "价格: " << coffee->getCost() << "\n";  // 8.0
```

### （3）适配器模式（Adapter）⭐⭐⭐⭐
- **核心**：让接口不兼容的两个类能够协作（转换器）。
```cpp
// 目标接口（调用方期望的）
class ILogger {
public:
    virtual void log(const std::string& msg) = 0;
    virtual ~ILogger() = default;
};

// 已有的第三方日志库（接口不兼容，无法修改）
class ThirdPartyLogger {
public:
    void writeLog(int level, const std::string& content) {
        std::cout << "[Level " << level << "] " << content << "\n";
    }
};

// 适配器：把 ThirdPartyLogger 适配成 ILogger
class LoggerAdapter : public ILogger {
public:
    explicit LoggerAdapter(std::shared_ptr<ThirdPartyLogger> logger)
        : logger_(std::move(logger)) {}

    void log(const std::string& msg) override {
        logger_->writeLog(1, msg);  // 转换调用方式
    }

private:
    std::shared_ptr<ThirdPartyLogger> logger_;
};

// 使用：调用方只依赖 ILogger 接口
std::unique_ptr<ILogger> logger =
    std::make_unique<LoggerAdapter>(std::make_shared<ThirdPartyLogger>());
logger->log("系统启动");  // [Level 1] 系统启动
```

## 3.行为型模式
### （1）观察者模式（Observer）⭐⭐⭐⭐⭐
- **核心**：一对多依赖，被观察者状态变化时自动通知所有观察者。
- **注意**：与“发布-订阅模式不同”
    - ![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260318165612748.png)
```cpp
#include <vector>
#include <algorithm>
#include <functional>

// 观察者接口
class IObserver {
public:
    virtual void onEvent(const std::string& event) = 0;
    virtual ~IObserver() = default;
};

// 被观察者（Subject）
class EventBus {
public:
    void subscribe(IObserver* observer) {
        observers_.push_back(observer);
    }

    void unsubscribe(IObserver* observer) {
        observers_.erase(
            std::remove(observers_.begin(), observers_.end(), observer),
            observers_.end()
        );
    }

    void publish(const std::string& event) {
        for (auto* obs : observers_) {
            obs->onEvent(event);  // 通知所有订阅者
        }
    }

private:
    std::vector<IObserver*> observers_;
};

// 具体观察者
class EmailNotifier : public IObserver {
public:
    void onEvent(const std::string& event) override {
        std::cout << "[邮件] 收到事件: " << event << "\n";
    }
};

class LogObserver : public IObserver {
public:
    void onEvent(const std::string& event) override {
        std::cout << "[日志] 记录事件: " << event << "\n";
    }
};

// 使用
EventBus    bus;
EmailNotifier email;
LogObserver   logger;

bus.subscribe(&email);
bus.subscribe(&logger);
bus.publish("用户下单");
// [邮件] 收到事件: 用户下单
// [日志] 记录事件: 用户下单
```

### （2）策略模式（Strategy）⭐⭐⭐⭐⭐
- **核心**：封装一族可互换的算法，消灭 if-else，运行时动态切换。
```cpp
#include <functional>
#include <map>

// 策略接口
class IPayStrategy {
public:
    virtual void pay(int amount) = 0;
    virtual ~IPayStrategy() = default;
};

// 具体策略
class AliPayStrategy : public IPayStrategy {
public:
    void pay(int amount) override {
        std::cout << "[支付宝] 支付 " << amount << " 元\n";
    }
};

class WxPayStrategy : public IPayStrategy {
public:
    void pay(int amount) override {
        std::cout << "[微信支付] 支付 " << amount << " 元\n";
    }
};

// Context：持有策略，对外提供统一接口
class PayContext {
public:
    void setStrategy(std::unique_ptr<IPayStrategy> s) {
        strategy_ = std::move(s);
    }

    void pay(int amount) {
        if (!strategy_) throw std::runtime_error("未设置支付策略");
        strategy_->pay(amount);
    }

private:
    std::unique_ptr<IPayStrategy> strategy_;
};

// 使用：运行时切换策略，无 if-else
PayContext ctx;
ctx.setStrategy(std::make_unique<AliPayStrategy>());
ctx.pay(100);  // [支付宝] 支付 100 元

ctx.setStrategy(std::make_unique<WxPayStrategy>());
ctx.pay(200);  // [微信支付] 支付 200 元
```

### （3）模板方法模式（Template Method）⭐⭐⭐⭐
- **核心**：父类定义算法骨架（流程），子类实现具体步骤，不改变整体结构。
```cpp
// 抽象基类：定义模板方法（算法骨架）
class DataProcessor {
public:
    // 模板方法：final 防止子类覆写流程
    void process() {
        readData();     // 步骤1：读数据
        processData();  // 步骤2：处理数据（子类实现）
        saveData();     // 步骤3：保存结果
    }

    virtual ~DataProcessor() = default;

protected:
    virtual void readData()    { std::cout << "[基类] 读取数据\n"; }
    virtual void processData() = 0;  // 纯虚函数，子类必须实现
    virtual void saveData()    { std::cout << "[基类] 保存数据\n"; }
};

// 具体子类：只需实现变化的部分
class CSVProcessor : public DataProcessor {
protected:
    void processData() override {
        std::cout << "[CSV] 解析 CSV 格式数据\n";
    }
};

class JSONProcessor : public DataProcessor {
protected:
    void processData() override {
        std::cout << "[JSON] 解析 JSON 格式数据\n";
    }
};

// 使用
std::unique_ptr<DataProcessor> p = std::make_unique<CSVProcessor>();
p->process();
// [基类] 读取数据
// [CSV] 解析 CSV 格式数据
// [基类] 保存数据
```

### （1）为什么需要熔断机制
- 因为线上最麻烦的故障，很多都不是“服务彻底挂了”，而是“**服务还活着，但已经很慢、很抖、半死不活**” ——> 这时候如果没有熔断，调用方会**继续把请求打过去**，结果就是**超时堆积、线程堆积、连接池耗尽**，最后从一个点故障变成**整条链路雪崩**。
- **状态机**：
    - Closed 正常放行
    - Open 直接拒绝该实例请求
    - Half-Open 放少量探测流量

## 1.虚函数指针存在对象的哪来？如何判断
- vptr 存在对象本身的内存里（**通常是对象的开头**），每个对象一份
- **判断方法"观察对象内存偏移"**：
```cpp
#include <iostream>
using namespace std;

class Foo {
public:
    int  a = 1;
    int  b = 2;
    virtual void f() {}
};

int main() {
    Foo obj;
    char* base = (char*)&obj;

    cout << "对象起始地址：" << (void*)base        << endl;
    cout << "&a 的偏移：   " << (char*)&obj.a - base << endl;  // 输出：8
    cout << "&b 的偏移：   " << (char*)&obj.b - base << endl;  // 输出：12
    // 偏移0~7 就是 vptr 占用的空间
    return 0;
}
```

## 2.malloc如何分配虚函数指针
- **⚠️ 核心问题**：malloc 只分配原始内存，**不调用构造函数**
    - vptr 不会被自动写入
    - 直接调用虚函数 = 未定义行为（崩溃）
- **✅ 正确方式**：malloc + placement new
```cpp
#include <iostream>
#include <cstdlib>
#include <new>       // placement new 需要
using namespace std;

class Animal {
public:
    int age;
    Animal(int a) : age(a) { cout << "构造，age=" << age << endl; }
    virtual ~Animal()       { cout << "析构" << endl; }
    virtual void speak()    { cout << "Animal speak, age=" << age << endl; }
};

int main() {
    // 1. malloc 分配原始内存（此时 vptr 未初始化！）
    void* raw = malloc(sizeof(Animal));
    if (!raw) return -1;

    // 2. placement new：在已分配的内存上调用构造函数（vptr 在这里写入）
    Animal* p = new(raw) Animal(5);
    //            ↑ placement new 语法

    // 3. 正常使用
    p->speak();           // ✅ 输出：Animal speak, age=5

    // 4. 必须手动调用析构函数（不能用 delete！）
    p->~Animal();         // ✅ 手动析构

    // 5. 释放内存
    free(raw);            // ✅ 用 free，不用 delete

    return 0;
}
```

## 9.缓存
### （1）为什么要做多级缓存
- 缓存本质上是在**拿一致性**和**复杂度**换 **延迟、吞吐和成本**
- 缓存最适合**读多写少**、**变化不频繁**、**原始存储较慢**、**网络距离远**的数据

### （2）什么情况下不适合使用缓存

#### <1>对一致性要求极高的场景
- 这类场景**最怕“读到旧数据”**
    - **库存扣减**：比如电商秒杀，库存刚减完，缓存里还是旧库存，就可能超卖。
    - **账户余额 / 钱包金额**：用户刚转账成功，缓存里还是老余额，会直接引起业务错误

#### <2>写远多于读，或者数据变化特别频繁
- 缓存最**适合“读多写少”** ——>如果数据一直在变，缓存会频繁失效、重建，收益很低，反而增加维护成本。
    - **实时位置数据**：比如司机位置、骑手位置，每几秒变一次。
    - **高频监控指标**：QPS、CPU、内存等秒级甚至毫秒级变化的数据。
    - **实时竞价 / 动态报价**：价格变化快，旧值几乎没有复用价值。
- 刚写进去就过期，**命中率很低**；不停更新缓存，**缓存层本身变成负担**

#### <3>数据安全或敏感性很高，不适合落入缓存
![](https://cdn.jsdelivr.net/gh/xianyubuxian-TXY/hexo-images/images/20260323214112982.png)

### （3）本地缓存、Redis、MySQL 各自优缺点
- **本地缓存**：最快，但最不一致
- **Redis**：共享缓存，性能高，适合大多数缓存场景
- **MySQL**：权威存储，最可靠，但最慢，不应该当高频缓存用

## 10.多级缓存（本地缓存、redis、mysql）

### （0）多级缓存数据一致性问题解决方案
#### <1> redis 与 mysql一致性
- **旁路缓存**：当写数据时，先将数据写入mysql，在删除其在redis中的缓存，之后读取该数据发送miss后，再从mysql中获取并写入redis缓存

#### <2> 本地缓存 与 redis缓存 一致性
- **local cache的ttl < redis cache的ttl**
- **配合消息队列MQ**：
    - 当写数据时，先通过**旁路缓存**，让redis删除缓存
    - 为了让local cache知道缓存失效，redis会向消息队列中发布消息，通知缓存失效（**发布-订阅模式**）
    - local cache收到缓存失效的通知后，删除本地缓存
    - 之后redis读取数据，miss后，从mysql读并将数据写入缓存后，redis向消息队列中发布消息，通知缓存生效，本地在从redis读取对应缓存存入到local cache


### （1）多级缓存的常见结构
- 本地缓存、Redis、Mysql
```bash
请求
  ↓
L1: 本机进程内缓存（每台应用机器自己一份）
  ↓ miss
L2: 共享分布式缓存（Redis / Memcached / SQL cache）
  ↓ miss
DB / 对象存储 / 搜索引擎（权威数据源）

写入链路常见是：
DB 提交成功
  ↓
失效/更新 L2
  ↓
广播或订阅事件，失效所有机器的 L1

```

## 12.函数返回局部变量字符串问题
### （1）代码示例1
- **常见误区**：const T& 不是能延长生命周期吗？
    - 是的，但 **只对直接绑定临时对象（prvalue）有效**：
```cpp
const string& func() {
    string a="Hello World";        // 局部变量，栈上分配
    return a;        // 返回局部变量的引用
}                    // ← a 在这里被析构，内存被释放！

const string& b = func();  // b 绑定的是一块已被释放的内存
cout << b << endl;          // 💥 未定义行为（Undefined Behavior）
```
- **正确写法**：
```cpp
string func() {          // 返回值，不是引用
    string a = "hello";
    return a;            // 触发 NRVO（编译器直接在调用方构造，零拷贝）
}

string b = func();       // ✅ 安全，b 拥有自己的数据
cout << b << endl;       // ✅ 正常输出 "hello"
```
