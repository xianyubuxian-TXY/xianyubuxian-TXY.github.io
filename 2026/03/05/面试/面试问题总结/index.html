（3）可执行文件的装入与执行全过程（重点）

  • 系统中要运行用户程序，就必须先将其装入内存中
  • 需要明确：可执行文件内部已经是二进制数据了

<1>二进制指令执行过程（基础知识）

  • 指令执行过程：取指、译码、执行、访存、写回（流水线加速）
1）取指

2）译码
  • 把取指阶段拿到的「二进制指令码」翻译成CPU能理解的”操作命令”：
    • 指令的二进制编码包含两部分核心信息：操作码（做什么） + 操作数（对谁做）；
    • 译码阶段会解析出：要执行的操作（比如加法、减法、移动数据）、操作数的来源（比如寄存器、内存地址、立即数）。
3）执行
  • 执行译码阶段解析出的具体操作，是指令的”计算核心阶段”：
    • 如果是算术运算（加减乘除）、逻辑运算（与或非）：由运算器完成计算；
    • 如果是数据移动（比如把A寄存器的值移到B寄存器）：直接在寄存器间传输；
    • 如果是控制操作（比如跳转）：修改程序计数器（PC）的值，改变指令执行顺序。
4）访存
  • 与内存交互：读取内存中的数据，或把执行结果写入内存（不是所有指令都需要这一步）：
    • 读内存：比如指令要求”把内存地址0x100的值加载到寄存器”，执行阶段算出地址后，访存阶段从内存取数；
    • 写内存：比如指令要求”把ALU的计算结果存入内存地址0x200”，访存阶段完成写入；
    • 无内存操作的指令（比如寄存器间的加法）：这一步会直接跳过，仅做”空操作”。
5）写回
  • 把指令执行的最终结果写回CPU的寄存器（或特殊存储位置），供后续指令使用：
    • 如果是算术运算：把ALU的结果写入目标寄存器；
    • 如果是读内存：把从内存取出的数据写入目标寄存器；
    • 写回是指令执行的”收尾”，确保结果被保存，后续指令能调用。

<2>ELF可执行目标文件 格式解析

  • 先明确：ELF可执行文件，本质是一个文件，存在磁盘中
  • ELF头： ELF头中的e_entry给出了程序的起始虚拟地址（入口点）（通常是_start,而不是main），即执行程序时第一条指令的地址
  • 程序头表：在后面我们会看到，ELF可执行文件的“只读（代码）段”、“可读写（数据）段”会被映射到进程的虚拟地址空间的“只读代码段”、“可读写数据段” ——> 这个映射关系就是由“程序头表”记录

<3>可执行文件的存储器映像

  • 进程的“虚拟地址空间”布局大体固定但地址可随机化，而其中的“只读代码段”、“可读写数据段”则是从ELF可执行文件中装入
  • 注意：程序加载过程中，实际上并没有真正从磁盘上加载代码和数据到内存，而是仅仅创建了只读代码段和可读写数据段对应“VMA”、“页表” ——> 只有在执行代码过程中发生了“缺页”异常时，才真正从磁盘加载代码和数据到主存。

<4>可执行文件的装入与执行全过程

0）映射关系
  • 虚拟地址空间 ——> ELF可执行文件空间 （VMA：虚拟内存区域）
  • 虚拟地址空间 ——> 物理地址（内存空间）（页表）
  • 进程中指令、数据的地址，都是虚拟地址，访问时，会通过页表获取其映射的物理地址（内存地址），当发现没有相应映射时，就会发生缺页异常，需要从磁盘中加载对应的指令/数据，那从磁盘的哪里加载呢？ ——> 前面我们说过，虚拟地址空间的“只读代码段”、“可读写数据段”都要从ELF可执行文件中加载，所以会访问内存中的VMA，通过“虚拟地址空间”获取其映射的“ELF可执行文件空间”，进而从对应的ELF可执行文件中加载指令/数据
1）全景概览

  • 可执行文件载入到运行过程：
    • 1.解析ELF文件头：从“ELF头”中解析出“起始虚拟地址”，存入到“程序计数器PC”中；从“程序头表”中解析出“ELF文件空间”与“虚拟地址空间”的映射（“只读代码段”、“可读写数据段”）
    • 2.创建虚拟地址空间：创建“VMA”，记录“虚拟地址空间”到“ELF文件空间”的映射关系；创建“页表”，记录“物理空间（内存空间）”到“虚拟地址空间”的映射关系
      • 注意：此时只是建立了映射关系，但还没有将“代码”、“数据”加载到内存中
    • 3.执行指令：
      • 取指：“程序计数器PC”中记录了下一条指令的地址，初始时已经被设置为了“起始虚拟地址”，程序执行时，会访问PC中记录指令地址获取指令 ——> 因为记录的是“虚拟地址”，所以会访问“页表”（有快表就先访问快表）获取其对应的“物理地址”，如果不存在，则发生缺页异常，进而访问“VMA”，获取“虚拟地址”对应的“ELF可执行文件地址”，从磁盘文件将指令加载到内存中，然后填写页表映射；
      • 译码、执行
      • 访存：译码后的指令中的数据地址也是“虚拟地址”，“获取内存数据”、“将执行结果写回内存”都要经历类似“取指”的过程
      • 写回：把指令执行的最终结果写回CPU的寄存器（或特殊存储位置），供后续指令使用

三、网络与IO

1.介绍一下epoll多路复用

（0）一句面试版

（1）什么是epoll？

  • epoll 是 Linux 内核提供的一种高效的 I/O 多路复用机制，用于监控多个文件描述符（fd）上的 I/O 事件。
  • 它是 select 和 poll 的增强版本，能够在大量并发连接中高效工作。

（2）epoll的核心API（三个系统调用）

  • epoll_wait 不一定阻塞
    • timeout = -1：一直阻塞
    • timeout = 0：立即返回（非阻塞轮询）
    • timeout > 0：最多阻塞 timeout 毫秒

（3）epoll的工作流程

  • 通过epoll_create创建唯一标识epoll内核事件表的文件描述符
  • 创建要监听的文件描述符的epoll_event进行绑定，并设置监听的事件（读事件/写事件/异常事件等）
  • 通过epoll_ctl向epoll内核事件表中注册要监听的文件描述符对应的epoll_event
  • 调用epoll_wait等待内核事件表上事件的发生，根据返回的“就绪文件描述符”数量，从出参epoll_event数组中取出文件描述符一次进行处理

（4）epoll的两种工作模式

<1>LT（Level Triggered）— 水平触发（默认）

  • 缓冲区有数据就一直通知
  • 可以不一次性读完，下次还会通知
  • 编程简单，不容易丢数据
  • 性能较低

<2>ET（Edge Triggered）— 边缘触发

  • 只在状态变化时通知一次
  • 必须一次性读完所有数据（必须循环读/写（以及 accept）直到返回 EAGAIN，否则可能丢后续通知。）（循环读到 EAGAIN）
  • 必须配合非阻塞 I/O 使用
  • 性能更高，减少了 epoll_wait 的调用次数

2.介绍一下Reactor模型

（1）Reactor模型是什么

  • Reactor 是一种事件驱动的网络编程模型。
    • 事件驱动：程序不是按固定顺序一直往下执行，而是先“等事件”，事件发生后再执行对应处理逻辑。
      核心思想是：
    • 用 select/poll/epoll 统一监听大量连接的 I/O 事件
    • 哪个连接就绪，就把事件分发给对应的 工作线程 去处理

（2）常见三种线程模型

（3）Reactor的工作流程（以epoll为例）

（4）与Proactor的区别

  • Reactor：通知你“可以读/写了”，read/write 由应用自己做
  • Proactor：通知你“读/写已经完成了”，I/O 操作由内核/系统异步完成后回调

💡 三者关系

（1）封装

  • 将数据和操作数据的方法绑定在一起，对外隐藏实现细节，只暴露接口。
  • 优点：
    • 保护数据安全（不能随意修改 balance）
    • 降低耦合度
    • 便于修改内部实现而不影响外部

（2）继承

  • 子类继承父类的属性和方法，实现代码复用。

<1>多重继承时的 vtable

<2>多继承需要解决的两个关键问题

1）名冲突
  • 在多继承中，当多个基类中包含同名的成员时，在派生类中就会出现名冲突
  • 解决方法：基类名受限访问
2）重复继承（菱形继承） —— 虚基类
  • 在多继承中，如果继承的类有公共的基类，则会出现重复继承 ——> 公共基类中的数据成员在多继承的派生类中有多个拷贝
  • 解决方案：虚基类
  • 通常情况下，我们需要把这两个x合并为一个x，此时，应把类A定义为类B和类C的虚基类，这样定义后，类D就只有一个成员x了
拓展：包含虚基类的类的“两个注意点”
  • 虚基类的构造函数由该类的构造函数直接调用
  • 虚基类的构造函数优先于非虚基类的构造函数执行

（3）多态

  • 同一接口，不同实现。分为编译时多态和运行时多态。
    • 编译时多态（静态多态）：函数重载 & 模板
    • 运行时多态（动态多态）：通过虚函数实现
  • 动态多态：基类通过virtual定义虚函数后，子类继承该基类并override虚函数，再通过基类指针/引用 指向/引用 子类对象

🎯 面试速答

  • 封装：将数据和方法绑定，通过”访问控制符”隐藏实现细节，只暴露必要接口。
  • 继承：子类继承父类的属性和方法，实现代码复用和层次结构。
  • 多态：同一接口不同实现。编译时多态通过”函数重载/模板”实现；运行时多态通过”虚函数和继承”实现，基类指针可调用派生类方法。

3.虚函数相关面试问题

（1）介绍一下虚函数、虚函数表、虚函数指针，以及三者如何协作实现”虚函数动态绑定”

  • 虚函数： virtual定义的成员函数
  • 虚函数表：对于每一个类，如果它有虚函数（包括从基类继承来的），则编译程序将会为其创建一个虚函数表（vtbl），表中记录了该类中所有虚函数的入口地址
    • 虚函数表存储在只读数据段 .rodata中
  • 虚函数表指针：当创建一个包含虚函数的类的对象时，在所创建对象的内存空间中有一个隐藏的指针（vptr）指向该对象所属类的虚函数表。
    • vptr初始化时机： 在构造函数执行时设置（先指向父类 vtable，再指向子类 vtable）
  • 虚函数动态绑定的实现方式：
    • 基类通过virtual定义成员函数，子类继承该基类后，由于包含虚函数（包括从基类继承来的），故程序会在 .rodata 段为该类创建一个虚函数表，为该类的所有对象共用。
    • 子类的虚函数表中，最开始存储的是父类的虚函数的入口地址，当子类override该虚函数后，虚函数表中对应位置将改为存储子类override后的虚函数的入口地址
    • 当通过引用/指针访问类的虚函数时，将会利用它们引用或指向的对象内存空间中的虚函数表指针从虚函数表中获取相应虚函数的入口地址来调用相应的函数

（2）vptr初始化时机

  • 在“构造函数”的初始化列表之后、函数体之前被隐式设置。
  • 先指向父类 vtable，再指向子类 vtable

<1>为什么在“初始化列表之后、函数体之前”

  • 在构造函数中可能调用虚函数

<2>为什么“先指向父类 vtable，再指向子类 vtable”

  • 因为构造 Derived 对象（子类对象）时，先执行 Base 构造函数（此时 vptr 指向 Base::vtable），再执行 Derived 构造函数（vptr 更新为 Derived::vtable）。

（3）为什么析构函数往往是虚函数，而构造函数不是？

<1>析构函数往往是虚函数

1）delete 操作的本质
  • 非虚析构：delete p 时，编译器根据指针类型 Base* 在编译期直接生成调用 Base::~Base() 的代码，不会查虚函数表。
  • 虚析构：析构函数地址存入 vtable，delete p 时通过 vptr → vtable → 析构函数地址，运行时找到对象实际类型的析构函数（Derived::~Derived()）进行调用，然后自动调用父类的析构函数。
  • 这就是为什么有继承关系且可能通过基类指针删除对象时，往往使用虚析构函数
拓展1：子类析构后自动调用父类析构的原理
  • 📌 关键澄清：这个行为与”虚”无关！ 无论析构函数是否为虚函数，子类析构后都会自动调用父类析构。
拓展2：子类析构函数的自动重写机制


<2>构造函数不能使虚函数

  • 在创建对象时，会自动调用构造函数，如果将构造函数设置为虚函数，则会通过虚函数表指针去虚函数表中获取函数入口地址并执行，但一个对象的虚函数表指针是在构造函数中进行初始化的（指向虚函数表），也就意味着将用“未初始化的虚函数表指针（还未执行对应虚函数表）”去虚造构造入口地址，进而导致错误

（5）纯虚函数 & 抽象类

  • 抽象类：含有纯虚函数的类
    • 抽象类不能实例化，子类必须实现纯虚函数
