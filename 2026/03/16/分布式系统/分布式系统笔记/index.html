一、分布式系统

1.分布式系统概述

  • 分布式系统 = 多台机器通过网络协作，对外表现为一个统一系统。
  • 它是一个广泛的技术概念，涵盖计算、存储、数据库、消息、缓存、协调等多个领域。
  • 分布式带来了性能、容量、可靠性的提升，但也引入了网络不可靠、数据一致性、部分失败等核心挑战，CAP 定理和 BASE 理论是理解这些挑战的理论基础。

二、微服务

  • 微服务是分布式的一种应用形式，但分布式远不止微服务

1.微服务介绍

  • 微服务的本质：
    • 逻辑上独立的服务，部署在不同的进程/机器上，通过 gRPC（或其他协议）进行网络通信。
    • 微服务 = 拆成独立进程 + 通过网络通信协作，RPC 是通信的核心技术。
    • RPC参考：protobuf的使用 中 “RPC（远程过程调用）”
  • 微服务的底层原理：
    • 把原来同一进程内的函数调用，变成了跨网络的远程调用。
    • RPC（尤其是 gRPC）是实现这种远程调用的主流方式——它通过 Stub 自动生成 + Protobuf 序列化 + HTTP/2 传输，让远程调用看起来和本地调用一样简单。
    • 但微服务通信不只有 RPC，还有消息队列（异步）、REST（对外 API）等，实际项目中是混合使用的。

2.实现微服务项目的两种方式

  • 后面都将以：gRPC分布式框架实现的微服务用户管理系统为例（包含auth、user独立服务）

（1）单仓库，部署时选择性注册

  • 在同一个项目中通过gRPC框架实现所有服务，然后在不同的服务器上运行该项目，但只是注册的服务不同

（2）独立仓库，各服务完全独立（更标准的微服务做法）

  • 独立 ≠ 孤立,微服务模块之间要通信

补充：独立 ≠ 孤立

  • 独立指的是：独立开发、独立部署、独立扩展、独立内存空间。
  • 不是指它们互不往来——恰恰相反，微服务之间的通信是整个架构的核心。
  • 示例：用户下单购买商品

3.微服务独立仓库下的公共代码复用问题

  • 在”独立仓库“中，各服务完全独立，但每个服务都需要：数据库连接、日志、配置加载、gRPC 拦截器、错误处理、Proto 生成代码……如果每个仓库都复制一份，维护噩梦。
  • 解决方案：把公共代码抽成独立的共享库。

（1）哪些代码会重复

（2）四种主流解决方案

<1>共享库（最常用）

  • 把公共代码抽成一个独立的包，各服务通过包管理器引入。

<2>Monorepo（单仓库多服务）

  • 所有服务放在同一个仓库，但各自独立构建、独立部署。

<3>代码生成 / 脚手架模板

  • 用模板工具自动生成每个新服务的骨架代码。

<4>Sidecar / 基础设施下沉（Service Mesh）

  • 把公共功能从代码中彻底移出，变成独立的基础设施。
